 Resumo
O TrendMart é um sistema de análise de dados de grande escala desenvolvido para suporte à decisão num contexto de e-commerce, tendo como base o dataset público Olist, que contém dados reais de transações, avaliações e comportamento de navegação do mercado brasileiro.
O sistema foi concebido para servir seis perfis de analistas de dados — Gestor Executivo, Analista de Vendas, Analista de Marketing, Gestor de Customer Experience, Analista de Tendências e Data Scientist — cada um com necessidades analíticas distintas, num total de 22 requisitos funcionais e 6 requisitos não funcionais levantados por análise de cenário.
A arquitetura adota um Data Lake com organização em três camadas: Bronze, para preservação imutável dos dados originais; Silver, para limpeza e validação; e Gold, para estruturas analíticas otimizadas para consulta. Os dados são gerados por um simulador de eventos, ingeridos via Apache Kafka com Change Data Capture por Debezium, processados em Apache Spark, persistidos em formato Apache Iceberg e expostos através de vistas Trino. O processo de integração é orquestrado pelo Apache Airflow com cadência horária.
A camada analítica disponibiliza seis dashboards interativos desenvolvidos em Plotly Dash, um por perfil, com atualização automática a cada cinco minutos. O sistema inclui ainda um pipeline de machine learning diário que treina um modelo de previsão de procura por categoria de produto, disponibilizando previsões para os sete dias seguintes com métricas de desempenho associadas.
A reprodutibilidade total do sistema é garantida por Docker Compose, sem dependências externas ao código-fonte e ao dataset Olist.

palavras-chave: Medallion Architecture, Hive metastore, Change Data Capture, Trino, Apache Iceberg, Data Lakehouse

1. Definição do Sistema

1.1 Contexto de Aplicação
O Projeto baseia-se na simulação de uma plataforma de e-commerce de dimensão real, contendo informação sobre vendas, clientes, produtos, regiões geográficas e avaliações de encomendas. O contexto operacional do sistema simula uma organização de retalho online com presença nacional no Brasil, com dezenas de milhares de produtos distribuídos por mais de 70 categorias distintas, e com operações cobrindo os 27 estados brasileiros. Esta organização lida diariamente com volumes elevados de dados provenientes de fontes heterogéneas e com naturezas distintas: dados transacionais de encomendas registados em base de dados relacional, eventos comportamentais de sessões de utilizador em tempo real (clickstream), e avaliações textuais submetidas pelos clientes após a entrega dos produtos.
A complexidade analítica deste cenário resulta da combinação de três fatores. Primeiro, a heterogeneidade das fontes de dados: dados estruturados de vendas, dados semi-estruturados de eventos de navegação e dados não estruturados de texto de avaliações, cada um com volume, velocidade e natureza técnica distintos. Segundo, o requisito de análise temporal contínua: a detecção de tendências emergentes exige que os dados estejam disponíveis para análise de forma regular e atualizada, sem depender de extrações manuais ou processos batch de execução rara. Terceiro, a necessidade de suporte à decisão multidimensional: necessidades de análise distintas e complementares que o sistema deve satisfazer de forma coerente.
Para responder a estes desafios, o implementa uma arquitetura de Data Lakehouse com padrão Medallion (Bronze → Silver → Gold), integrando tecnologias de processamento distribuído, ingestão em tempo real, orquestração de pipelines e aprendizagem automática, num ambiente completamente containerizado e reproduzível. O sistema gera continuamente dados simulados com base no dataset Olist e processa-os através de um pipeline automático que os transforma progressivamente em informação analítica de valor.

1.2 Motivação e Objetivos do Trabalho
As organizações de retalho online que operam com análises baseadas em extrações manuais com latência de dias ou semanas, impede a detecção atempada de tendências. Este projeto substitui este modelo por um pipeline contínuo e automatizado que transforma dados heterogêneos em suporte à decisão para múltiplos perfis.
Integrar três fontes heterogêneas num pipeline unificado e tolerante a falhas. Estruturar os dados em camadas de qualidade crescente (Bronze/Silver/Gold) com rastreabilidade completa. Calcular métricas quantitativas de tendência. Produzir previsões diárias de procura por categoria. Expor os resultados num dashboard interativo com atualização automática de 5 em 5 minutos.

1.3 Análise da Viabilidade do Processo
Viabilidade técnica: Todas as ferramentas adotadas são de código aberto, maduras e amplamente utilizadas em produção: Apache Spark, Kafka, Iceberg, Trino, Airflow e MLflow têm adoção documentada em empresas como Netflix, LinkedIn e Uber. Nenhuma integração envolve tecnologia experimental.


Viabilidade operacional: O sistema é completamente reprodutível a partir do código-fonte. Um único comando (python Extra/start.py) inicializa os doze serviços Docker, cria a estrutura de dados e inicia o pipeline. O processo completo de arranque demora menos de 15 minutos numa máquina com 16 GB de RAM.
Viabilidade económica: O custo de operação do sistema é praticamente nulo. Todo o software utilizado é open-source e gratuito. A infraestrutura corre integralmente numa única máquina de desenvolvimento desktop de uso corrente serve como ambiente completo de produção.
Viabilidade de dados: A organização dispõe de dados operacionais contínuos provenientes das três fontes identificadas, registros transacionais de compras, eventos de sessão dos utilizadores e avaliações de clientes acumulados ao longo de vários anos de operação. O volume histórico disponível é suficiente para suportar todos os requisitos analíticos.

1.4 Recursos e Equipa de Trabalho
Recursos Humanos: O projeto foi desenvolvido por uma equipa de quatro estudantes de Engenharia e Ciência de Dados. Todos os membros participaram activamente em todas as fases do projecto desde a configuração da infraestrutura até ao desenvolvimento do dashboard adaptando o seu envolvimento às necessidades de cada etapa. Esta abordagem garantiu que toda a equipa adquiriu conhecimento transversal do sistema e que nenhuma componente ficou dependente de um único elemento. 
Recursos Materiais: O sistema corre integralmente em Docker Compose, numa única máquina de desenvolvimento. Os requisitos mínimos são 16GB de RAM, 4 núcleos de CPU e 20 GB de armazenamento SSD. Todo o software é open-source e gratuito.





1.5 Plano de Execução do Projeto
O desenvolvimento seguiu uma abordagem iterativa orientada a camadas, na ordem natural do fluxo de dados da infraestrutura.

O plano de execução do projeto está organizado em nove fases sequenciais e parcialmente sobrepostas. A primeira fase corresponde ao levantamento e análise de requisitos, bem como à definição da arquitetura global do sistema. Segue-se a configuração do ambiente de infraestrutura, assegurando que todos os serviços e dependências técnicas estejam operacionais antes do início do desenvolvimento. A terceira fase contempla a implementação das fontes de dados e dos mecanismos de geração e publicação de eventos. Com as fontes activas, procede-se à construção da camada de ingestão, responsável por recolher e persistir os dados em bruto. A quinta fase compreende a transformação e limpeza dos dados ingeridos, produzindo uma camada estruturada e validada. Segue-se o desenvolvimento do modelo analítico dimensional e a exposição de vistas de consulta para consumo pelos utilizadores. As fases sete e oito decorrem em paralelo: o desenvolvimento dos modelos de aprendizagem automática e a construção da interface de visualização e análise. Por fim, a nona fase abrange a redação do relatório e a documentação técnica do sistema.

2. Levantamento e Análise de Requisitos

O desenvolvimento de um sistema de análise de dados de grande escala exige, antes de qualquer decisão técnica, uma compreensão clara de quem vai usar o sistema e para quê. Sem esta ancoragem, corre-se o risco de construir uma infraestrutura tecnicamente sofisticada mas incapaz de responder às perguntas de negócio que justificam a sua existência. O levantamento de requisitos é, por isso, o primeiro passo estruturante do projeto no momento em que se traduzem necessidades de negócio em especificações concretas que guiam todas as decisões de arquitetura e implementação subsequentes.

O sistema serve uma organização de e-commerce com múltiplas áreas funcionais, cada uma com perspectivas, objetivos e ritmos de trabalho distintos. Em empresas reais, estas necessidades analíticas estão tipicamente distribuídas entre dois tipos de sistemas de informação: sistemas de que gerem dados operacionais e transacionais como vendas, receita e inventário, e sistemas que centralizam dados de comportamento de cliente, interações, feedback e satisfação. O Projeto replica analiticamente os dados que estes dois tipos de sistemas geram, consolidando-os numa camada analítica unificada.

A interface que concretiza este serviço é o dashboard desenvolvido pela equipa `data_analytics`. Esta equipa é técnica e única mas o produto que entrega ramifica-se em seis abas, cada uma orientada a um perfil de utilizador distinto. 
Os perfis de natureza ERP: Gestor Executivo, Analista de Vendas e Analista de Tendências  consomem dados transacionais e operacionais: receita, volume de vendas, crescimento e anomalias. 
Os perfis de natureza CRM: Analista de Marketing, Gestor de Customer Experience e Data Scientist consomem dados comportamentais e relacionais: funil de conversão, sentimento de cliente e previsão de procura. É esta ramificação que serve de estrutura ao levantamento de requisitos deste capítulo: cada aba corresponde a um perfil, cada perfil tem perguntas de negócio próprias, e cada pergunta origina requisitos funcionais concretos que o sistema tem de satisfazer.

Este capítulo organiza-se em três secções: o método de levantamento adotado (2.1), a caracterização dos seis perfis com os respetivos requisitos funcionais e não funcionais (2.2), e a validação de cada requisito contra a implementação real do sistema (2.3).




2.1 Método Adotado
O método adotado foi a análise de cenário orientada a perfis de utilizador. Para cada um dos seis perfis identificados, foram definidas as perguntas de negócio que o sistema deve ser capaz de responder, os dados necessários para o efeito, e as restrições operacionais específicas ao seu contexto de uso. Os requisitos funcionais derivam diretamente dessas perguntas, um requisito está satisfeito se e só se a pergunta que lhe corresponde pode ser respondida com o sistema construído.
Esta abordagem tem três vantagens metodológicas. Primeiro, força a ancoragem dos requisitos em necessidades concretas de negócio em vez de capacidades técnicas abstratas. Segundo, facilita a priorização  funcionalidades partilhadas por múltiplos perfis têm maior prioridade do que as específicas a um único perfil, o que guiou as decisões de arquitetura no sentido de maximizar o valor entregue com os recursos disponíveis. Terceiro, torna a validação dos requisitos sistemática  na secção 2.3, cada requisito é verificado contra a implementação real do sistema, tornando explícita a correspondência entre o que foi especificado e o que foi construído.
Para além dos requisitos funcionais por perfil, foram identificados requisitos não funcionais transversais relativos a latência, independência, rastreabilidade, escalabilidade e reprodutibilidade que condicionam a arquitetura e não são específicos a nenhum perfil de utilizador.

2.2 Organização dos Requisitos Levantados

2.2.1	Perfil Gestor Executivo
O Gestor Executivo é responsável por monitorizar a saúde global do negócio e tomar decisões estratégicas com base em indicadores de desempenho de alto nível. O seu interesse não está no detalhe operacional — está na evolução da receita, no volume de encomendas e no crescimento período a período. Trabalha com horizontes temporais de 90 dias e necessita que os dados estejam atualizados com frequência suficiente para detetar tendências antes que se tornem problemas.

Perguntas de negócio
Qual foi a receita total, o número de encomendas, os clientes únicos e o valor médio por encomenda no período?
Como evoluiu a receita e o volume de encomendas diariamente ao longo dos últimos 90 dias?

Requisitos funcionais derivados
RF1.1 — KPIs globais diários: receita total, volume de encomendas, clientes únicos e valor médio por encomenda, em série temporal de 90 dias.
RF1.2 — Evolução diária da receita e do volume de encomendas em formato de série temporal.

2.2.2	Perfil Analista de Vendas
O Analista de Vendas é responsável por avaliar o desempenho comercial da plataforma ao nível do detalhe — por categoria de produto e por região geográfica. O seu interesse está em perceber quais as categorias que mais contribuem para a receita, como se distribui geograficamente a base de clientes, e qual a composição da receita entre produto e frete. Trabalha com horizontes de 90 dias e necessita de vistas desagregadas que permitam identificar oportunidades e áreas de melhoria no portfólio de produtos. Este perfil é de natureza ERP — consome dados transacionais com granularidade por categoria, região e estado.

Perguntas de negócio
Quais são as categorias de produto com maior contribuição para a receita?
Como se distribui a receita por região e por estado?
Como evoluiu a receita ao longo do tempo com granularidade diária?
Qual a decomposição da receita entre valor de produto e custos de frete por categoria?

Requisitos funcionais derivados
RF2.1 — O sistema deve disponibilizar as top 10 categorias por receita total num horizonte de 90 dias.
RF2.2 — O sistema deve disponibilizar a distribuição de receita por macrorregião e por estado.
RF2.3 — O sistema deve disponibilizar a evolução temporal da receita com granularidade diária, numa janela de 90 dias.
RF2.4 — O sistema deve disponibilizar a decomposição de receita em valor de produto e valor de frete por categoria e por período.

2.2.3	Perfil Analista de Marketing
O Analista de Marketing é responsável por monitorizar o comportamento de navegação dos utilizadores na plataforma e avaliar a eficácia do funil de conversão. O seu interesse está em perceber em que etapa do funil os utilizadores abandonam, como se distribui o comportamento por tipo de dispositivo, como evoluem as sessões e utilizadores ativos ao longo do tempo, e se o comportamento difere entre dias úteis e fins de semana. Trabalha com horizontes de 7 dias para análise do funil e 30 dias para análise de padrões semanais. Este perfil é de natureza CRM — consome dados comportamentais gerados pelo clickstream.

Perguntas de negócio
Qual é o volume de eventos em cada etapa do funil de conversão e onde ocorre maior abandono?
Como se distribui o comportamento de navegação por dispositivo — mobile, desktop e tablet?
Como evoluiu o número de sessões e utilizadores ativos nos últimos 7 dias?
O comportamento de navegação difere entre dias úteis e fins de semana?

Requisitos funcionais derivados
RF3.1 — O sistema deve disponibilizar o funil completo de eventos — desde session_start até order_placed — com contagens por etapa.
RF3.2 — O sistema deve desagregar os eventos por tipo de dispositivo de acesso.
RF3.3 — O sistema deve disponibilizar a evolução temporal dos top 5 eventos mais frequentes com granularidade diária numa janela de 7 dias.
RF3.4 — O sistema deve disponibilizar o número de sessões e utilizadores únicos por dia.
RF3.5 — O sistema deve comparar o volume de eventos e sessões entre dias úteis e fins de semana numa janela de 30 dias.

2.2.4   Perfil Gestor de Customer Experience
O Gestor de Customer Experience é responsável por monitorizar a qualidade da experiência do cliente após a compra, nomeadamente através da análise das avaliações e do sentimento expresso. O seu interesse está em identificar padrões de insatisfação — categorias com ratings baixos, regiões com avaliações negativas recorrentes, deterioração temporal do sentimento — antes que se traduzam em perda de clientes. Trabalha com horizontes de 90 dias e necessita de vistas que permitam detetar precocemente sinais de degradação da experiência. Este perfil é de natureza CRM — consome dados de feedback e satisfação de cliente.

Perguntas de negócio
Como se distribui o sentimento das avaliações entre positivo, neutro e negativo?
Como evoluiu o rating médio ao longo do tempo?
Quais as categorias com maior volume de reviews e qual o seu rating médio?
Como se distribui o sentimento por região geográfica?

Requisitos funcionais derivados
RF4.1 — O sistema deve disponibilizar a distribuição de reviews por sentimento — positivo, neutro e negativo — com base na classificação do rating.
RF4.2 — O sistema deve disponibilizar a evolução do rating médio diário numa janela de 90 dias.
RF4.3 — O sistema deve disponibilizar as top 10 categorias por volume de reviews com o respetivo rating médio.
RF4.4 — O sistema deve disponibilizar a distribuição de sentimento por macrorregião geográfica.

2.2.5 Perfil Analista de Tendências
O Analista de Tendências é responsável por identificar padrões de crescimento e aceleração da procura no comportamento das vendas. Ao contrário do Analista de Vendas, que olha para o desempenho absoluto, este perfil foca-se na variação relativa — crescimento semana-sobre-semana e aceleração dessa taxa de crescimento. É o perfil mais diretamente alinhado com o objetivo central do enunciado: a deteção precoce de tendências de produto. Trabalha com horizontes de 60 dias e necessita de métricas de variação temporal com granularidade semanal. Este perfil é de natureza ERP — consome dados transacionais agregados com métricas derivadas de séries temporais.

Perguntas de negócio
Qual foi o crescimento da receita e do volume de encomendas semana-sobre-semana?
A taxa de crescimento está a acelerar ou a desacelerar?
Existem anomalias no padrão de crescimento que sinalizem comportamentos atípicos?
Quais as categorias com maior e menor crescimento médio nos últimos 30 dias?

Requisitos funcionais derivados
RF5.1 — O sistema deve calcular e disponibilizar o crescimento semana-sobre-semana da receita e do volume de encomendas.
RF5.2 — O sistema deve calcular e disponibilizar a aceleração da procura — variação da taxa de crescimento entre semanas consecutivas.
RF5.3 — O sistema deve detetar e assinalar anomalias no padrão de crescimento semanal, numa janela de 60 dias.
RF5.4 — O sistema deve disponibilizar o crescimento médio por categoria numa janela de 30 dias.

2.2.6 Perfil Data Scientist
O Data Scientist é responsável por interpretar os resultados dos modelos preditivos treinados pelo pipeline de machine learning e monitorizar a qualidade das previsões ao longo do tempo. O seu interesse está na disponibilidade das previsões de procura por categoria para os próximos sete dias e nas métricas de desempenho do modelo que as suportam. Ao contrário dos restantes perfis, não consome apenas dados de negócio — consome também métricas de avaliação de modelos que lhe permitem aferir a fiabilidade das previsões antes de as utilizar em decisões operacionais. Este perfil é de natureza CRM — as previsões de procura derivam de padrões comportamentais e transacionais dos clientes.

Perguntas de negócio
Qual é a previsão de encomendas para os próximos 7 dias por categoria de produto?
Qual o desempenho preditivo do modelo em termos de RMSE e MAE por categoria?

Requisitos funcionais derivados
RF6.1 — O sistema deve treinar um modelo de previsão de procura por categoria e persistir as previsões para os sete dias seguintes com as métricas de desempenho associadas.
RF6.2 — O sistema deve disponibilizar as previsões de procura e as métricas do modelo num dashboard interativo, atualizadas a cada execução diária do pipeline ML.

2.2.7 Requisitos Não Funcionais

Os requisitos não funcionais identificados são transversais a todos os perfis e condicionam as decisões de arquitetura do sistema.
RNF1 — Latência analítica: os dados disponibilizados no dashboard devem refletir os eventos gerados pelo simulador com uma latência máxima de uma hora, correspondente à cadência do pipeline batch de transformação Silver → Gold. O dashboard atualiza as suas vistas de cinco em cinco minutos.
RNF2 — Idempotência: qualquer operação de processamento ou carga deve poder ser repetida sem duplicar ou corromper dados. Este requisito aplica-se a todos os jobs Spark de transformação Silver (via operação MERGE INTO Iceberg), ao pipeline Gold (via modo append incremental com watermark) e às tabelas ML (via createOrReplace a cada execução diária).
RNF3 — Rastreabilidade: os dados originais devem ser preservados em formato imutável na camada Bronze do Data Lake, independentemente de qualquer transformação posterior. Em caso de erro na camada Silver ou no pipeline Gold, deve ser possível reprocessar os dados originais sem contactar novamente a fonte.
RNF4 — Reprodutibilidade: o sistema deve ser completamente reproduzível a partir do código-fonte e do dataset Olist público, sem dependência de dados externos não disponíveis. A reprodutibilidade é garantida pelo Docker Compose e pelo dataset Olist carregado na inicialização do simulador.
RNF5 — Escalabilidade horizontal: a arquitetura deve suportar aumento do volume de dados sem alteração estrutural do pipeline. A adoção do Apache Spark como motor de processamento garante que os jobs de transformação escalam horizontalmente com a adição de executores, sem alteração do código.
RNF6 — Orquestração auditável: a execução dos pipelines de dados deve ser gerida por um orquestrador com suporte a dependências entre tasks, retentativas automáticas e registo do estado de execução de cada run. Este requisito fundamenta a escolha do Apache Airflow.

3. Implementação do Sistema de Análise

3.1 Apresentação Geral
O TrendMart é um sistema de análise de dados em grande escala orientado ao suporte à decisão no domínio do retalho de e-commerce. O seu principal objectivo é simular, ingerir, transformar e analisar continuamente dados produzidos por uma plataforma digital de comércio electrónico, convertendo eventos operacionais heterogéneos numa infraestrutura analítica integrada capaz de suportar exploração de negócio, monitorização operacional e análise preditiva.
O sistema foi concebido para reproduzir não apenas os dados normalmente gerados por uma plataforma de e-commerce moderna, mas também os próprios padrões arquiteturais utilizados em ecossistemas reais de dados distribuídos. Neste contexto, o pipeline integra mecanismos de streaming de eventos, captura de alterações transacionais (CDC), armazenamento object-based, processamento distribuído com Apache Spark, tabelas analíticas Apache Iceberg, consultas SQL via Trino e orquestração contínua através do Apache Airflow.
A arquitetura implementa quatro capacidades principais. A primeira corresponde à ingestão contínua de dados em tempo quase-real. Eventos de navegação, transações de compra e avaliações de clientes são produzidos continuamente pelo simulador e capturados através de três mecanismos de ingestão distintos, cada um adaptado às características operacionais da respectiva fonte. A segunda capacidade corresponde à preparação e governação de dados. Os dados brutos provenientes das diferentes fontes são progressivamente transformados em representações limpas, tipadas, deduplicadas e semanticamente enriquecidas através de uma arquitetura em múltiplas camadas baseada no padrão Medalhão. A terceira capacidade corresponde à análise descritiva e exploratória. Sobre as tabelas analíticas produzidas pelo pipeline é construído um modelo dimensional em estrela, posteriormente exposto através de vistas SQL optimizadas para consumo por dashboards interativos e ferramentas externas de Business Intelligence. A quarta capacidade corresponde à análise preditiva. Modelos de Machine Learning treinados sobre os dados históricos do sistema produzem previsões de procura por categoria, disponibilizadas como artefactos analíticos integrados no ecossistema de consumo.

Desta forma, cada equipa pode evoluir internamente os seus componentes sem afectar directamente as restantes equipas, desde que preserve o contrato acordado.
A arquitectura adopta assim um modelo de desacoplamento funcional inspirado em plataformas modernas de Data Engineering, onde as diferentes fases do pipeline operam de forma independente e substituível. Equipa de Machine Learning
A equipa de Machine Learning é responsável pela componente preditiva do sistema. O seu principal objectivo consiste em utilizar os dados históricos produzidos pela camada Gold para treinar modelos de previsão de procura capazes de estimar as tendências futuras de encomendas por categoria de produto, para um horizonte temporal de sete dias.

3.2 Fontes de Dados
Na ausência de uma plataforma de e-commerce real em produção, o TrendMart recorre a um simulador contínuo desenvolvido especificamente para o projeto, capaz de reproduzir o comportamento operacional de uma loja digital moderna. O simulador gera continuamente eventos de navegação, transações de compra e avaliações de clientes, permitindo alimentar o pipeline analítico com dados persistentes, correlacionados e temporalmente consistentes.
Embora os dados sejam sintéticos, a sua geração é baseada no dataset real Olist Brazilian E-Commerce (https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce), utilizado como referência para produtos, categorias, vendedores, clientes, preços e avaliações. Esta abordagem permite preservar coerência semântica entre as diferentes fontes produzidas. Os produtos visualizados durante a navegação correspondem efetivamente a produtos existentes no catálogo; as compras resultam dessas interações; e as avaliações referem-se às encomendas efetivamente concluídas. O sistema evita assim a geração arbitrária de eventos independentes, aproximando-se mais do comportamento observado em plataformas reais de e-commerce.
A definição das três fontes de dados foi guiada por três objetivos arquiteturais deliberados.
O primeiro objetivo foi a heterogeneidade estrutural. As fontes diferem intencionalmente tanto no formato dos dados como no mecanismo de ingestão utilizado: eventos semi-estruturados em streaming, registos transacionais estruturados e ficheiros de texto livre armazenados em object storage. Esta diversidade impossibilita uma abordagem puramente relacional e justifica a adopção de uma arquitectura como, Data Lake, por exemplo, com múltiplas camadas de transformação.
O segundo objectivo foi a simulação de dados contínuos e de elevado volume, particularmente ao nível do clickstream de navegação. Apesar de o simulador não conseguir correr tempo suficiente para produzir volumes verdadeiramente massivos de eventos, toda a arquitetura foi concebida para suportar cenários de ingestão contínua e crescimento progressivo dos dados ao longo do tempo. A geração contínua de eventos permite reproduzir padrões típicos de streaming observados em sistemas reais de observabilidade e análise comportamental, possibilitando validar mecanismos de ingestão incremental, processamento contínuo e particionamento temporal.
O terceiro objectivo consistiu em aproximar o sistema dos padrões operacionais encontrados em plataformas reais de e-commerce, procurando utilizar soluções tecnológicas modernas para a sua implementação. Para além da simulação dos dados de negócio, foram reproduzidos mecanismos de integração amplamente utilizados em arquitecturas de dados atuais, como streaming de eventos via Kafka, CDC em bases de dados e ingestão assíncrona em object storage. A arquitectura analítica recorre ainda a ferramentas modernas como Apache Spark para processamento distribuído, Apache Iceberg para gestão de tabelas em data lake e Trino para consulta interactiva e federada. O sistema procura assim representar não apenas os dados, mas também uma stack tecnológica realista e alinhada com práticas atuais de engenharia de dados.

Tabela 11- Fontes de Dados do Sistema (Resumo) 

3.2.1 Dados Clickstreams de Navegação

Os clickstream de navegação representam a principal fonte de observabilidade comportamental do sistema. Cada evento descreve uma interação realizada por um utilizador durante uma sessão na plataforma, permitindo reconstruir percursos completos de navegação, abandono, exploração de categorias e conversão em compra.
As sessões são geradas pelo módulo session.py através da função simulate_session(), responsável por modelar o percurso do utilizador segundo um funil probabilístico de conversão. Durante a execução de cada sessão, os eventos são produzidos sequencialmente e publicados em tempo real no tópico Kafka clickstream_events.
A publicação utiliza o session_id como chave de partição, garantindo que todos os eventos pertencentes à mesma sessão permanecem na mesma partição Kafka e preservam a sua ordem temporal relativa. Esta decisão é particularmente importante para processamento downstream, uma vez que permite reconstruir corretamente o percurso cronológico do utilizador sem necessidade de reordenação adicional.

Esta dualidade permite que o funil completo seja analisado diretamente a partir do clickstream, mantendo a integridade transacional na base de dados relacional.

Ruído
Para além da geração normal de eventos, o sistema introduz perturbações controladas através do módulo noise.py, com o objectivo de aproximar os dados produzidos das imperfeições observadas em ambientes reais.
Entre as anomalias simuladas encontram-se ausência de session_id (falhas de rastreamento), valores inválidos no campo device (bots, crawlers ou instrumentação incorrecta) e eventos sem event_type definido (falhas de tracking). 
A introdução deliberada de ruído permite validar a robustez dos pipelines de ingestão e transformação, evitando assumir condições idealizadas de qualidade de dados.

Armazenamento Inicial
Assim, os eventos de navegação no pipeline encontram-se inicialmente no tópico Kafka clickstream_events. Cada evento é serializado em formato JSON e armazenado como mensagem UTF-8, utilizando o session_id como chave de partição e o payload do evento como valor.
O Kafka funciona como uma camada de ingestão e transporte de eventos em tempo real, armazenando as mensagens de forma sequencial por partição e offset. 

3.2.2 Dados de Vendas

As vendas representam a componente transaccional estruturada do sistema. Uma compra concluída desencadeia duas acções em simultâneo: o evento order_placed é publicado no tópico Kafka clickstream_events como qualquer outro evento do funil, e em paralelo é gerado um registo de compra persistido no schema relacional do PostgreSQL através do módulo db_writer.py. Esta dualidade permite que o funil completo seja analisado diretamente a partir do clickstream, mantendo simultaneamente uma fonte transaccional normalizada para análise de receita e dimensões.

Armazenamento Inicial
Os pedidos são armazenados num modelo relacional composto por cinco tabelas na terceira forma normal (3NF):
Customers: um registo por cliente único; contém localização geográfica.
Sellers: um registo por vendedor único.
Products: um registo por produto único; contém categoria e número de fotografias.
Orders: tabela de factos da transacção; customer_id é chave estrangeira para customers; session_id permite correlacionar a compra com o percurso de navegação no clickstream.
Order_items: linha de item da ordem; referência orders, products e sellers por chaves estrangeiras.


Figura 5- Modelo Relacional da Fonte de Dados de Vendas (PostgreSQL) 

Processo de Escrita
Uma compra só é gerada quando o funil de clickstream atinge a etapa order_placed — ou seja, quando o utilizador completa o checkout com sucesso. Nesse momento, a função simulate_session() devolve simultaneamente os eventos de clickstream (publicados no Kafka) e um dicionário purchase que é persistido no PostgreSQL. 
Cada compra desencadeia uma sequência de cinco operações geridas atomicamente numa única transacção pelo módulo db_writer.py: 
Upsert em customers — insere o cliente com estado e cidade; se o customer_id já existir (cliente recorrente), a operação é descartada silenciosamente (ON CONFLICT DO NOTHING).
Upsert em sellers — idêntico ao passo anterior para o vendedor.
Upsert em products — idêntico para o produto, incluindo categoria e número de fotografias.
Insert em orders — cria o registo de facto da compra, referenciando o customer_id já garantido no passo 1.
Insert em order_items — cria o item da ordem, referenciando order_id, product_id e seller_id.

O campo session_id na tabela orders preserva a ligação à sessão de navegação, permitindo a correlação downstream entre o trajecto comportamental no clickstream e a conversão efetiva na fonte transaccional.

Ruído
A fonte transaccional introduz anomalias controladas que afectam aproximadamente 19% dos registos: preços negativos ou nulos no campo price, códigos de estado mal formatados em state (diferenças de capitalização e caracteres adicionais) e categoria inválida em category.
Adicionalmente, o mecanismo de at-least-once delivery do Debezium é simulado através de noop_update_purchase(), que executa UPDATE orders SET state = state WHERE order_id = ? sem alterar qualquer valor — gerando um segundo evento CDC com op='u' para o mesmo order_id. Estes duplicados são posteriormente resolvidos na camada Silver através de deduplicação MERGE INTO sobre Apache Iceberg.





3.2.3 Avaliações de Clientes
As avaliações de clientes constituem a fonte textual semi-estruturada do sistema e representam a componente menos rígida do ecossistema de dados do TrendMart.
Ao contrário das restantes fontes, as avaliações são produzidas de forma desacoplada e assíncrona através do módulo review_writer.py, que associa reviews a aproximadamente 40% das ordens concluídas. A geração ocorre com um atraso aleatório entre 30 e 300 segundos após a compra, simulando um intervalo entre recepção do produto e submissão de feedback pelo cliente.

Estrutura do Ficheiro
O conteúdo das avaliações consistem exclusivamente em texto livre em português, sem utilização de campos delimitados ou schemas explícitos. Os únicos metadados estruturados encontram-se codificados no nome do ficheiro, obrigando o pipeline downstream a combinar processamento textual com extração de metadados contextuais.

3.3 Área de Preparação de Dados
A área de preparação de dados constitui a componente intermédia do pipeline responsável por transformar dados operacionais heterogêneos em estruturas analíticas consistentes e semanticamente enriquecidas. O seu papel é actuar como camada de mediação entre os mecanismos de ingestão das fontes e o modelo dimensional consumido pelas equipas analíticas e de machine learning.
No contexto do TrendMart, esta área recebe continuamente três tipos de dados estruturalmente distintos — eventos semi-estruturados de navegação, alterações transacionais capturadas por CDC e ficheiros não estruturados de texto livre — e converte-os progressivamente em tabelas analíticas com schema fixo, tipagem forte e garantias de consistência transaccional.
A implementação desta área é da responsabilidade da Equipa de Engenharia de Dados, cujo contrato operacional estabelece explicitamente os limites da sua intervenção no pipeline. Como entrada, a equipa consome exclusivamente dados brutos provenientes das três fontes do simulador; como saída, entrega tabelas Silver limpas, duplicadas e semanticamente normalizadas, prontas a suportar a construção da camada Gold.
A preparação de dados encontra-se organizada segundo uma arquitectura incremental em múltiplas camadas, onde cada etapa introduz um novo nível de estrutura, qualidade e significado analítico. Esta separação permite desacoplar ingestão, preservação histórica, validação e transformação semântica, reduzindo dependências directas entre mecanismos de captura de dados e consumo analítico downstream.


3.3.1 Justificação da Abordagem
A área de preparação de dados é implementada segundo uma arquitectura Data Lakehouse baseada no padrão Medalhão (Bronze / Silver / Gold), utilizando armazenamento object-based sobre MinIO, tabelas Apache Iceberg e processamento distribuído com Apache Spark.
O modelo Lakehouse combina duas propriedades tradicionalmente separadas em arquitecturas de dados. Por um lado, herda do Data Lake a capacidade de receber dados heterogéneos no seu formato original sem necessidade de schema rígido na entrada. Por outro, incorpora características típicas de Data Warehouse, incluindo schema enforcement, garantias ACID, controlo transacional e capacidade de consulta analítica via SQL.
No TrendMart, esta separação manifesta-se explicitamente através das diferentes camadas do pipeline. A camada Bronze corresponde à componente “lake”, preservando os dados no estado mais próximo possível do original. As camadas Silver e Gold correspondem à componente “house”, onde os dados passam a obedecer a schemas explícitos, regras de qualidade, tipagem forte e estruturas optimizadas para consumo analítico.
A adopção desta arquitectura resulta directamente da heterogeneidade das fontes utilizadas pelo sistema. O pipeline recebe simultaneamente eventos JSON publicados em Kafka, envelopes CDC produzidos pelo Debezium e ficheiros de texto livre armazenados em object storage. Estas fontes apresentam níveis distintos de estrutura, granularidade e estabilidade de schema, tornando inviável a ingestão directa para um modelo relacional rígido sem uma etapa intermédia de normalização.
Um sistema puramente relacional exigiria schemas definidos à entrada e forte homogeneidade estrutural entre fontes, condições incompatíveis com a variabilidade do envelope CDC e com o caráter não estruturado das avaliações textuais. A arquitectura Lakehouse resolve este problema ao desacoplar ingestão e estruturação semântica: os dados entram primeiro no Bronze sem imposição de schema analítico e apenas posteriormente são validados, tipados e enriquecidos na camada Silver.
A separação progressiva em camadas introduz ainda vantagens arquitecturais importantes.
A primeira é a rastreabilidade histórica. Como os dados originais permanecem preservados no Bronze, qualquer alteração futura às regras de transformação pode ser aplicada retroativamente através de reprocessamento, sem necessidade de re-ingestão das fontes.
A segunda é a independência operacional. Os jobs Spark Structured Streaming mantêm checkpoints persistentes com o histórico dos ficheiros já processados, permitindo reiniciar pipelines após falhas sem risco de duplicação de dados.
A terceira é a separação de responsabilidades entre ingestão e qualidade de dados. Os consumers de ingestão limitam-se a preservar eventos no Bronze, sem conhecimento das regras semânticas aplicadas posteriormente na Silver. Esta independência reduz o acoplamento entre componentes e torna as diferentes etapas do pipeline evolutivamente substituíveis.


3.3.2 Camada Bronze

A camada Bronze constitui o ponto de entrada de todos os dados no Data Lakehouse e representa a componente de preservação histórica do sistema. O seu objectivo não é transformar, validar ou enriquecer dados, mas sim garantir que todos os eventos produzidos pelas fontes são armazenados de forma persistente, rastreável e o mais próxima possível do estado original.
Esta camada funciona como a fonte de verdade imutável do pipeline. Independentemente das regras de qualidade aplicadas posteriormente, os dados originais permanecem sempre disponíveis para auditoria, reprocessamento ou reconstrução de tabelas analíticas futuras.
Os únicos campos adicionados durante a ingestão correspondem a metadados operacionais de rastreabilidade:
ingested_at, correspondente ao instante de recepção do registo;
source, identificando o mecanismo de ingestão utilizado (kafka, debezium ou minio).
Nenhuma transformação semântica é aplicada nesta fase. Esta decisão é deliberada: impor regras de qualidade demasiado cedo aumentaria o acoplamento entre ingestão e processamento analítico, dificultando reprocessamentos futuros e reduzindo flexibilidade arquitetural.
A existência da camada Bronze desacopla ainda dois ritmos distintos do sistema: o ritmo de armazenamento dos dados e o ritmo de processamento analítico. Os mecanismos de ingestão podem continuar a escrever continuamente no Bronze independentemente da disponibilidade ou estado dos jobs Silver, tornando as duas etapas operacionalmente independentes.
	A análise de qualidade dos dados brutos, realizada sobre a camada bronze antes de qualquer transformação, revelou problemas transversais às três fontes de dados: encomendas em PostgreSQL, eventos de clickstream em Parquet e avaliações em ficheiros de texto.

Na fonte de encomendas, cerca de 17% dos registos apresentam pelo menos um problema de qualidade, o que resulta numa taxa de validade de apenas 83%. O problema mais frequente é o formato incorreto do campo de estado geográfico, que afeta aproximadamente 6,5% do total, seguido por cerca de 5,9% de registos com espaços extra no nome da categoria, o que compromete a correspondência correta com outras tabelas e pode distorcer análises agregadas. Acresce que aproximadamente 4,2% das encomendas têm preço igual ou inferior a zero, violando uma regra fundamental de negócio e tornando inválido qualquer cálculo de receita baseado nesses registos sem prévia correção.
No clickstream, a taxa de validade sobe para 90%, mas persistem três tipos de problemas relevantes. O mais significativo é a presença de cerca de 4,2% de eventos gerados por dispositivos inválidos como bots, crawlers e televisões, que introduzem tráfego não humano e podem inflar métricas como visualizações de produto e sessões iniciadas. Aproximadamente 3,2% dos eventos não têm session_id, o que impossibilita a reconstrução de percursos de navegação e a análise de funis de conversão nesses casos. Adicionalmente, cerca de 2,3% dos eventos carecem de event_type, tornando impossível classificar a ação realizada pelo utilizador.
As avaliações em texto são a fonte com mais tipos distintos de problemas. Quase 7% dos ficheiros são duplicados, e a grande maioria desses duplicados partilha o mesmo identificador de avaliação com outro ficheiro, o que pode levar a contagem dupla na análise de sentimento. Para além disso, uma pequena percentagem de registos não tem identificador de avaliação, de cliente ou de encomenda, tornando-os total ou parcialmente inúteis para cruzamento com outras fontes. Existem ainda avaliações com rating fora do intervalo válido de 1 a 5, representando cerca de 4,3% dos casos, e uma pequena fração de ficheiros com corrupção de encoding por conversão incorreta de UTF-8 para Latin-1, produzindo caracteres ilegíveis no texto das mensagens.
Em conjunto, estas anomalias justificam a existência de uma camada de limpeza Silver, onde preços inválidos, categorias mal formatadas, estados com formato errado, eventos de bots e duplicados de avaliações devem ser tratados antes de qualquer processamento analítico ou de aprendizagem automática.



Dados de clickstream de navegação

Origem
Os eventos de clickstream têm origem no módulo simulator/producer.py, que publica cada evento no tópico Kafka clickstream_events em formato JSON. O Kafka atua como buffer durável entre o simulador e o consumer de ingestão, garantindo que nenhum evento se perde mesmo que o consumer esteja temporariamente indisponível.

Extração e Transporte
Para extrair e transportar os eventos o módulo ingestion/consumer.py subscreve o tópico Kafka e consome as mensagens em modo contínuo. Cada mensagem JSON é desserializada e os seus campos são escritos diretamente no Bronze — à excepção do campo properties, que é um objecto JSON aninhado cuja estrutura varia consoante o event_type. Como o Parquet não suporta objetos de estrutura variável, o campo é serializado de volta para string Json antes de ser escrito. A desserialização fica reservada para a camada Silver, onde o Spark aplica um schema fixo via from_json. 

Armazenamento
Os eventos são escritos em ficheiros Parquet no sub-bucket bronze/clickstream/, com particionamento hierárquico em quatro níveis: year, month, day, hour. O schema fixo do ficheiro contém os campos event_id, session_id, user_id, event_type, timestamp, device, properties e ingested_at. 

Fonte 2 — Ordens de Compra

Origem
As ordens de compra começam escritas no modelo relacional do PostgreSQL Amazon Sales (customers, sellers, products, orders, order_items).
 
Extração
O Debezium monitoriza continuamente o Write-Ahead Log (WAL) do PostgreSQL e captura cada mutação das tabelas orders e order_items — INSERT, UPDATE ou snapshot inicial — publicando-as nos tópicos Kafka debezium.public.orders e debezium.public.order_items, respectivamente, encapsuladas num envelope JSON com os campos before, after, op e ts_ms. Este mecanismo de Change Data Capture permite capturar a totalidade do ciclo de vida de cada registo sem qualquer alteração ao schema ou ao código do simulador.
As tabelas de dimensão customers, sellers e products não são capturadas pelo CDC — o Debezium monitoriza exclusivamente orders e order_items. Os atributos relevantes de dimensão (estado, categoria) já viajam desnormalizados dentro desses registos, e as dimensões formais são construídas mais tarde na camada Gold a partir dos dados Silver. 
 
Transporte
O módulo ingestion/cdc_consumer.py subscreve os dois tópicos Debezium (debezium.public.orders e debezium.public.order_items) e desembrulha o envelope CDC para extrair o campo after, encaminhando cada mensagem para o buffer correspondente (orders_buf ou order_items_buf) com base no campo source.table. São processadas as operações c (INSERT), u (UPDATE) e r (snapshot inicial); os eventos d (DELETE) são descartados porque o modelo de dados do simulador não produz eliminações de ordens. O campo cdc_ts_ms é preservado no registo Parquet para que a camada Silver consiga ordenar os eventos e desduplicar pelo mais recente em caso de múltiplos UPDATEs ao mesmo order_id.  A inclusão dos UPDATEs no Bronze é intencional: o simulador executa uma operação UPDATE sem alteração de dados (noop_update) para simular o comportamento de at-least-once delivery do Debezium, gerando um segundo evento com o mesmo order_id; se esses UPDATEs fossem descartados no Bronze, o mecanismo de deduplicação MERGE INTO da camada Silver não teria oportunidade de actuar.
 
Armazenamento
Os registos são escritos em ficheiros Parquet nos sub-buckets bronze/orders/ e bronze/order_items/, com particionamento hierárquico em quatro níveis: year, month, day, hour. O schema de orders contém order_id, customer_id, session_id, purchase_timestamp, state, cdc_ts_ms e ingested_at; o de order_items contém order_item_id, order_id, product_id, seller_id, price, freight_value, category, cdc_ts_ms e ingested_at. 


Fonte 3 — Avaliações de Clientes

Origem
As avaliações estão escritas como ficheiros de texto no bucket MinIO raw-reviews/. Cada ficheiro tem o nome {review_id}_{order_id}.txt — os únicos metadados estruturados associados à avaliação estão codificados no nome do ficheiro, não no seu conteúdo.

Extracção e transporte
O módulo ingestion/file_watcher.py inspeciona o bucket raw-reviews/ a cada 30 segundos via API S3 compatível com MinIO. Para cada ficheiro .txt ainda não processado — identificado por comparação com o conjunto de object_keys registados em .watcher_state.json — o conteúdo completo é lido e incluído como campo raw_content no registo Parquet, preservando o texto livre intacto para extracção de informação na camada Silver. Após a escrita do batch, o object_key de cada ficheiro processado é adicionado ao ficheiro de estado, garantindo que o mesmo ficheiro nunca é ingerido duas vezes mesmo após reinicialização do processo.

Armazenamento
As avaliações são escritas em ficheiros Parquet no sub-bucket bronze/reviews/, com particionamento em três níveis: year, month, day — um nível a menos do que as outras duas fontes, dado o volume mais reduzido e a ausência de granularidade horária no mecanismo de polling. O schema de cada ficheiro contém os campos file_path, review_id, order_id, raw_content e ingested_at, sendo o raw_content o texto livre completo do ficheiro sem qualquer transformação. 

Métodos Comuns

Particionamento hierárquico no padrão Hive
O particionamento hierárquico constitui o principal mecanismo de optimização física utilizado na camada Bronze. Os ficheiros Parquet são organizados segundo a convenção Hive (campo=valor), reconhecida nativamente pelo Spark e pelo PyArrow.
O objectivo principal desta estratégia é permitir partition pruning durante a leitura. Quando um job Silver necessita apenas de dados pertencentes a uma determinada janela temporal, o Spark consegue eliminar antecipadamente todas as partições irrelevantes sem sequer abrir os respectivos ficheiros.
Esta otimização reduz significativamente operações de I/O, desserialização e scanning de metadata, melhorando diretamente o desempenho dos jobs downstream.
O clickstream e as ordens utilizam particionamento em quatro níveis (year/month/day/hour) devido ao maior volume e granularidade temporal contínua. As avaliações utilizam apenas três níveis (year/month/day), uma vez que o menor volume de dados não justifica subdivisão horária adicional.


Micro-batching
A ingestão na camada Bronze utiliza uma estratégia de micro-batching, onde múltiplos registos são acumulados temporariamente em memória antes de serem escritos em ficheiros Parquet únicos.
Esta abordagem procura equilibrar dois requisitos contraditórios.
Por um lado, ficheiros demasiado pequenos degradam o desempenho do Spark devido ao elevado overhead de metadata e abertura de ficheiros. Por outro, batches demasiado grandes aumentam a latência entre a geração do evento e a sua disponibilidade analítica.
Os consumers Kafka e CDC resolvem este equilíbrio através de dois critérios alternativos de flush:
escrita imediata após 500 registos;
Escrita após 30 segundos sem flush.
O primeiro critério optimiza eficiência de leitura sob carga elevada; o segundo garante baixa latência durante períodos de tráfego reduzido.
O file watcher das avaliações segue um modelo ligeiramente distinto: cada ciclo de polling produz diretamente um batch correspondente ao conjunto de novos ficheiros detectados no bucket MinIO.


3.3.3 Camada Silver

A camada Silver representa o ponto de transição entre armazenamento bruto e estrutura analítica governada. É nesta etapa que os dados deixam de ser apenas artefactos persistentes e passam a obedecer explicitamente a regras de qualidade, schemas analíticos e garantias de consistência transaccional.
Enquanto a camada Bronze privilegia preservação e rastreabilidade, a camada Silver privilegia consistência semântica e fiabilidade analítica.
Cada job Silver lê os ficheiros Parquet correspondentes da camada Bronze, aplica validações específicas ao domínio de dados respectivo e escreve o resultado em tabelas Apache Iceberg registadas no catálogo lake.
É nesta camada que se concentram:
validação de schema;
tipagem forte;
normalização semântica;
enriquecimento contextual;
deduplicação;
filtragem de ruído;
garantias ACID sobre escrita incremental.
As tabelas produzidas nesta etapa constituem a base operacional consumida posteriormente pela Equipa de Engenheiros Analíticos para construção do modelo dimensional Gold.





Job 1 — silver_clickstream.py

Entrada
Ficheiros Parquet de bronze/clickstream/, lidos com schema explícito para evitar inferência automática.

Transformações e regras de qualidade
O campo properties é desserializado para struct com schema definido, extraindo product_id, category, price, state, city, query e reason. Destes, apenas category, product_id e price são promovidos a colunas individuais na saída; state e city são fundidos num único campo location via coalesce (estado prioritário sobre cidade); query e reason são descartados.
Os eventos com event_id, session_id ou event_type nulos ou vazios são descartados, assim como os que não produzem um event_ts válido após conversão do timestamp.
Os valores do campo device são normalizados para o vocabulário controlado {mobile, desktop, tablet}: valores não reconhecidos são normalizados para "unknown". 

Armazenamento
Tabela Iceberg lake.silver.clickstream com schema fixo e tipagem forte, em modo append. O schema da tabela contém os campos event_id, session_id, user_id, event_type, event_ts (timestamp tipado), device, category, product_id, price, location e ingested_at. 

Job 2 — silver_orders.py

Entrada
Dois conjuntos de ficheiros Parquet: bronze/order_items/ (stream principal — contém price, freight_value, product_id, seller_id e category) e bronze/orders/ (leitura em batch por foreachBatch — contém customer_id, session_id, purchase_timestamp e state). Os dois DataFrames são joined em order_id antes de qualquer transformação.

Transformações e regras de qualidade
Os campos price e freight_value são convertidos de string para DOUBLE e os registos com price ≤ 0 ou freight_value < 0 são descartados.
O purchase_timestamp — enviado pelo Debezium em microssegundos desde epoch — é convertido para Timestamp via divisão por 1 000 000; registos com timestamp inválido são descartados.
O campo category é normalizado com TRIM para remover espaços excedentes.
O campo state é validado contra os 27 códigos de estado brasileiros: variantes com casing errado ("sp"), espaços (" SP ") ou typos ("SP0") resultam em state = NULL e region = "Desconhecido", sem descartar a ordem.
O total_value é calculado como round(price + freight_value, 2).
Cada ordem é enriquecida com o campo region, derivado do mapeamento dos 27 estados para as cinco macrorregiões brasileiras. 

Armazenamento
Tabela Iceberg lake.silver.orders. O schema contém os campos order_id, session_id, customer_id, product_id, seller_id, category, price, freight_value, total_value, purchase_ts, state, region e ingested_at. 





Job 3 — silver_reviews.py

Entrada
Ficheiros Parquet de bronze/reviews/, que contêm os campos review_id, order_id, raw_content e ingested_at. Os campos review_id e order_id foram extraídos do nome do ficheiro durante a ingestão Bronze — chegam já como colunas tipadas. 

Transformações e regras de qualidade
Registos com review_id ou order_id nulos ou vazios são descartados.
O rating é extraído do texto livre por duas expressões regulares — "Dou X estrelas em 5" e "Classifico este produto com X/5"; registos sem match válido no intervalo [1, 5] são descartados.
O campo message é criado por TRIM(raw_content); registos com mensagem vazia após o trim são descartados.
O text_length é calculado como contagem de palavras por split em espaços. 

Armazenamento
Tabela Iceberg lake.silver.reviews. O schema contém os campos review_id, order_id, rating (INT), message, text_length e ingested_at. 

Mecanismos Comuns

Checkpoint do Spark Structured Streaming
Os três jobs utilizam trigger(availableNow=True), que instrui o Spark a processar todos os ficheiros Bronze ainda não consumidos e a terminar — em vez de correr indefinidamente como um stream contínuo. O estado de progresso é mantido num checkpoint persistido em MinIO (s3a://silver/_checkpoints/{clickstream,orders,reviews}): o Spark regista neste checkpoint os ficheiros Bronze já processados em runs anteriores e, no início de cada novo run, lê apenas os ficheiros ainda não registados. Este mecanismo garante independência: se um job falhar a meio e for relançado pela DAG, não reprocesse dados já escritos na Silver.

Deduplicação com MERGE INTO Iceberg
Os jobs de ordens e reviews enfrentam um problema adicional de deduplicação: o Bronze pode conter múltiplos eventos para o mesmo order_id (devido ao at-least-once delivery do Debezium) ou para o mesmo review_id (devido às submissões duplicadas injectadas pelo simulador). A deduplicação é feita em dois níveis. Dentro de cada micro-batch processado pelo Spark, uma window function (ROW_NUMBER OVER PARTITION BY order_id/review_id ORDER BY cdc_ts_ms DESC) retém apenas o evento mais recente por chave. Entre runs, o MERGE INTO Iceberg com WHEN NOT MATCHED THEN INSERT * garante que um order_id ou review_id já existente na Silver nunca é duplicado por runs posteriores — independentemente do número de vezes que o mesmo evento apareça no Bronze. O job de clickstream não necessita deste mecanismo porque os eventos de clickstream são naturalmente únicos por event_id e são escritos em modo append directo.

O código dos três jobs é da responsabilidade da Equipa de Engenharia de Dados; a execução é agendada e despoletada pela DAG trendmart_gold_pipeline da Equipa de Engenharia de Infraestrutura (infrastructure/dags/dag_trendmart.py), com cadência horária definida como requisito de SLA pela Equipa de Engenharia Analítica.


3.4 Área de Dados do Sistema de Análise
A camada Gold constitui a componente analítica do pipeline, onde os dados previamente limpos na camada Silver são transformados num modelo estruturado e orientado à análise de negócio. O seu principal objetivo é garantir que os dados chegam aos consumidores já agregados, enriquecidos e organizados por dimensão analítica, eliminando qualquer necessidade de lógica de transformação no cliente.
A implementação desta camada é da responsabilidade da equipa de Engenharia Analítica. O seu contrato define a entrada como as tabelas Silver e a saída como um modelo analítico em esquema estrela, exposto sob a forma de tabelas Iceberg e vistas SQL em Trino.



3.4.1 Modelo Dimensional

O sistema de dados analítico do TrendMart implementa um modelo dimensional em esquema estrela na camada Gold do Data Lakehouse, armazenado em tabelas Apache Iceberg sobre MinIO e acessível via Trino. A escolha de um modelo dimensional — em detrimento de uma modelação relacional normalizada (3NF) — decorre dos padrões de acesso analítico dominantes: queries que atravessam múltiplas dimensões (por exemplo, "receita por categoria, por mês, por região"), contagens e agregações sobre grandes volumes de factos, e filtragens sobre atributos descritivos das dimensões. O esquema estrela minimiza o número de joins necessários por query analítica e torna as estruturas de dados imediatamente compreensíveis para analistas sem conhecimento técnico do pipeline. 

Figura 6 - Modelo Dimensional em Esquema Estrela (Camada Gold) 

O processo de conceção seguiu os quatro passos do método de Kimball, com as escolhas de design ancoradas nos requisitos funcionais do capítulo 2.

Passo 1 — Identificação dos Processos de Negócio
Foram identificados três processos de negócio distintos mas complementares, cada um correspondendo a uma tabela de factos:
Vendas e Encomendas: quanto vendemos, onde, e em que categorias? O ciclo de vida de uma encomenda desde a compra até à sua conclusão, com métricas financeiras — suporta os perfis Gestor Executivo (RF1.1–RF1.2), Analista de Vendas (RF2.1–RF2.4) e Analista de Tendências (RF5.1–RF5.4).
Comportamento de Navegação: como é que os utilizadores navegam e onde abandonam? Os eventos que os utilizadores geram na plataforma, que permitem analisar o funil de conversão digital — suporta o perfil Analista de Marketing (RF3.1–RF3.5).
Avaliações de Clientes: o que dizem os clientes sobre os produtos? O feedback qualitativo pós-compra, com classificação de sentimento — suporta o perfil Gestor de Customer Experience (RF4.1–RF4.4).


Passo 2 — Definição da Granularidade
Para a tabela de factos fact_sales, a granularidade é ao nível da ordem individual: cada linha representa uma transação de compra. Esta granularidade permite agregar métricas em qualquer nível superior (categoria, dia, região) sem perda de informação.
Para a tabela de factos fact_clickstream, a granularidade é ao nível do evento individual: cada linha representa um clique, visualização, ou ação no funil. Esta granularidade permite reconstruir sessões completas e calcular taxas de conversão em qualquer ponto do funil.
Para a tabela de factos fact_reviews, a granularidade é ao nível da avaliação individual: cada linha representa um ficheiro de review processado, com o rating, sentimento, categoria e região associados.

Passo 3 — Identificação das Dimensões
A seleção das dimensões partiu da análise dos eixos de análise exigidos pelos requisitos funcionais do capítulo 2. Os perfis de natureza ERP e o Gestor de Customer Experience necessitam de análise por categoria; o Analista de Vendas e o Gestor de Customer Experience necessitam de análise geográfica; o Analista de Marketing necessita de análise temporal enriquecida com atributos de calendário. Isto determina exactamente três dimensões, necessárias e suficientes:
dim_date: responde ao quando com atributos derivados. Criada como calendário completo de 2020 a 2030 (4 018 datas), com atributos year, month, quarter, week, day_of_week, is_weekend. É referenciada exclusivamente por fact_clickstream via date_id, justificada pelo RF3.5 — comparação do comportamento de navegação entre dias úteis e fins de semana. As tabelas fact_sales e fact_reviews utilizam as colunas de data diretamente (purchase_date e review_date), uma vez que nenhum requisito associado a essas fontes exige atributos de calendário derivados.
dim_category: responde ao o quê ao nível de categoria. Contém os nomes de categoria derivados das ordens e do clickstream, com category_id gerada por hash determinístico do nome. Partilhada pelas três tabelas de factos.
dim_geography: responde ao onde. Mapeia os 27 estados brasileiros para as cinco macrorregiões (Norte, Nordeste, Centro-Oeste, Sudeste, Sul), com geo_id gerado por hash do código de estado. Partilhada por fact_sales e fact_reviews.
Não foram criadas dimensões ao nível de produto ou de cliente porque nenhum requisito funcional do capítulo 2 exige análise a esse nível de detalhe — toda a análise de produto é feita ao nível de categoria (RF2.1, RF4.3, RF5.3) e toda a análise de cliente é expressa em contagens de clientes únicos (RF1.1, RF2.2), não em atributos de clientes individuais.


Passo 4 — Identificação dos Factos
A tabela de factos fact_sales contém métricas aditivas (price, freight_value, total_value), as chaves de negócio order_id e customer_id, e chaves estrangeiras para dim_category (category_id) e dim_geography (geo_id). O customer_id é preservado como atributo degenerado para permitir a contagem de clientes únicos (RF1.1, RF2.2). A data da compra é armazenada diretamente como purchase_date — coluna que serve simultaneamente como eixo temporal de análise e como coluna de particionamento da tabela Iceberg.
A tabela de factos fact_clickstream contém event_type e device como factos degenerados (vocabulários controlados com número fixo de valores), session_id e user_id como chaves de negócio, e chaves estrangeiras para dim_date (date_id) e dim_category (category_id). A FK para dim_date é a única no modelo — justificada pelo RF3.5, que exige comparação do comportamento de navegação entre dias úteis e fins de semana via o atributo is_weekend. Não tem FK para dim_geography porque nenhum requisito do perfil Analista de Marketing (RF3.1–RF3.5) exige análise geográfica do comportamento de navegação.
A tabela de factos fact_reviews contém rating (semi-aditivo — o indicador relevante é a média, não a soma), sentiment como facto degenerado (positive, neutral, negative), text_length como medida descritiva, e chaves estrangeiras para dim_category (category_id) e dim_geography (geo_id). O geo_id é resolvido em gold_reviews.py por join com silver.orders para obter o estado do cliente da ordem associada, seguido de lookup em dim_geography — garantindo que a geografia fica inscrita no próprio facto e não depende de um join posterior através de fact_sales. A data da avaliação é armazenada diretamente como review_date.
A tabela ml_demand_forecast é uma tabela Gold adicional que não faz parte do modelo dimensional principal mas partilha a convenção de nomenclatura e é acessível via Trino. É gerada pela DAG independente trendmart_ml_pipeline e consumida diretamente pelo dashboard sem passar por vistas.


3.4.2 Implementação Física do Schema

Os jobs Spark da equipa analytical_engineering são invocados pela DAG trendmart_gold_pipeline com cadência horária. Existem dois padrões de escrita distintos consoante o tipo de tabela.

Tabelas de Dimensões

O job de dimensões (gold_dimensions.py) recria integralmente as três tabelas — dim_date, dim_category e dim_geography — a cada execução via writeTo().createOrReplace().  Esta escolha é deliberada: as dimensões têm um conjunto de valores pequeno e estável, e recalculá-las completamente garante que qualquer nova categoria ou novo estado presente no Silver fica refletido imediatamente, sem lógica de merge ou detecção de diferenças.

Tabelas de factos

Os três jobs de factos (gold_sales.py, gold_clickstream.py, gold_reviews.py) usam um modelo de carga incremental baseado na janela de execução do Airflow. O DAG passa a cada job os parâmetros data_interval_start e data_interval_end — os timestamps exatos da hora que está a ser processada —, e o job filtra a tabela Silver pelo campo ingested_at dentro dessa janela:
ingested_at >= data_interval_start  AND  ingested_at < data_interval_end
O campo ingested_at é atribuído por-registo no momento em que cada consumer Bronze recebe a mensagem (do Kafka, do CDC ou do MinIO), pelo que reflecte o instante real de ingestão de cada evento individual — não o timestamp do ficheiro Parquet que o contém.

Na primeira execução, quando a tabela Gold ainda não existe, o job usa writeTo().createOrReplace() para a criar com o schema e particionamento corretos. Nas execuções seguintes usa writeTo().append(), adicionando apenas os registos da hora processada. Se não existirem registos na janela — por exemplo num período de tráfego nulo —, o job termina sem escrever, evitando commits vazios.
Este modelo é distinto do mecanismo incremental da camada Silver: os jobs Silver usam Spark Structured Streaming com trigger(availableNow=True) e um checkpoint persistido em MinIO, que rastreia quais ficheiros Bronze já foram lidos pelo path — sem dependência do Airflow. O Gold delega essa responsabilidade ao orquestrador, o que torna cada run Gold determinístico e reproduzível: re-executar a janela das 13:00–14:00 produz sempre o mesmo resultado, independentemente de quando a re-execução ocorre.

3.5 O Processo de Integração de Dados
 
O processo de integração de dados do TrendMart segue o paradigma ELT (Extract, Load, Transform): os dados são primeiro extraídos das fontes e carregados no Data Lake Bronze no seu formato original, e só depois transformados nas camadas Silver e Gold. Esta abordagem preserva os dados originais indefinidamente como fonte de verdade imutável, desacopla o ritmo de ingestão do ritmo de transformação e permite reprocessar toda a cadeia a partir do Bronze sem re-ingestão das fontes.
 
3.5.1 Modelo do Processo de Integração (BPMN)
 
O processo de integração de dados é modelado segundo a notação BPMN, com três pools sequenciais correspondentes às três camadas do Data Lake: o Pool Bronze, responsável pela ingestão contínua das fontes; o Pool Silver, responsável pela transformação e validação Bronze → Silver via Spark; e o Pool Gold, responsável pela integração no modelo dimensional e criação das vistas Trino. O diagrama completo encontra-se no Anexo A. 

Pool Bronze — Ingestão Contínua
O Pool Bronze agrupa três lanes paralelas, uma por fonte de dados, cada uma com um consumer dedicado que corre continuamente e de forma independente da orquestração Airflow. O consumer Kafka subscreve o tópico clickstream_events e acumula eventos em memória até ao limiar de 500 registos ou 30 segundos, escrevendo então um ficheiro Parquet na partição Bronze correspondente. O consumer CDC subscreve os dois tópicos Debezium (debezium.public.orders e debezium.public.order_items) e aplica o mesmo mecanismo de micro-batching às ordens de compra capturadas do PostgreSQL. O file watcher consulta o bucket raw-reviews/ a cada 30 segundos, identifica ficheiros novos por comparação com o estado persistido em .watcher_state.json, e escreve os conteúdos em Parquet Bronze. Os três processos correm em paralelo e não têm dependências entre si.

 
Pool Silver — Transformação Batch (Airflow + Spark)
O Pool Silver é orquestrado pelo Airflow com execução horária. As três tarefas Silver executam em paralelo, cada uma processando os ficheiros Bronze disponíveis desde o último ciclo através de Spark Structured Streaming com trigger(availableNow=True).
Cada job aplica as regras de validação, limpeza e enriquecimento descritas na secção 3.3.3 e escreve os resultados nas tabelas Iceberg Silver correspondentes. A idempotência é assegurada por dois mecanismos distintos: o silver_clickstream garante exactly-once através do sink nativo do Iceberg combinado com checkpoint Spark; os jobs silver_orders e silver_reviews usam foreachBatch com operação MERGE INTO para deduplicação cross-batch. 
 
Pool Gold — Carga Analítica (Airflow + Spark)
O Pool Gold inicia após a conclusão das três tasks Silver. A task gold_dimensions executa primeiro, recriando as três dimensões com createOrReplace. Após a sua conclusão, as tasks gold_sales, gold_clickstream e gold_reviews executam em paralelo, fazendo append incremental às tabelas de factos para a janela horária processada. Por fim, a task init_views actualiza as seis vistas Trino, tornando os dados imediatamente disponíveis para o dashboard.

3.5.3 Ferramentas
 
As ferramentas que compõem o pipeline foram seleccionadas em conjunto pela equipa de infraestrutura e pelas equipas utilizadoras de cada componente, com base em critérios de compatibilidade técnica entre si, maturidade em ambientes de produção e adequação ao volume e natureza de dados do sistema. Todas são open-source e correm integralmente em Docker Compose.
 
Apache Kafka
O Apache Kafka é a plataforma de streaming utilizada como buffer durável para os eventos de clickstream. A sua adopção justifica-se por três requisitos que alternativas mais simples não satisfazem em conjunto.
O primeiro é a tolerância a falhas do consumer. Sem Kafka, a opção mais directa seria o simulador escrever eventos directamente para MinIO — mas se o consumer estiver em baixo quando os eventos chegam, esses eventos perdem-se sem possibilidade de recuperação. O Kafka persiste as mensagens em disco independentemente do estado do consumer: quando o consumer reinicia, retoma a partir do último offset confirmado sem perda de dados.
O segundo é a ordenação garantida por sessão. A análise de funil depende de conhecer a sequência de eventos dentro de cada sessão — se o utilizador viu o produto antes de comprar, quanto tempo decorreu entre a visualização e o abandono do carrinho. Sem ordenação garantida, estas análises produzem resultados incorrectos. O Kafka garante ordenação dentro de cada partição, e usando session_id como chave de partição, todos os eventos de uma sessão chegam ao consumer na ordem em que foram gerados.
O terceiro é o desacoplamento entre produção e consumo. O simulador publica eventos ao seu próprio ritmo, independentemente de o consumer estar disponível ou de quão rapidamente processa. Sem este desacoplamento, uma lentidão ou falha no consumer bloquearia ou atrasaria a geração de dados.
As alternativas consideradas não satisfazem estes três requisitos em conjunto. O RabbitMQ é uma queue de mensagens tradicional adequada para task queues, mas apaga as mensagens após consumo — sem replay em caso de falha — e não tem ordenação por partition key. O Redis Streams é mais leve mas com persistência menos robusta e sem o conceito de partition key para garantir ordenação por sessão. A escrita directa para MinIO elimina o buffer e a tolerância a falhas do consumer, tornando a ingestão dependente da disponibilidade contínua de ambos os processos em simultâneo.
 
Debezium
O Debezium é o conector de Change Data Capture utilizado para capturar as alterações nas tabelas orders e order_items do PostgreSQL. Monitoriza o Write-Ahead Log (WAL) do PostgreSQL e publica cada INSERT, UPDATE ou snapshot inicial como um evento estruturado nos tópicos Kafka debezium.public.orders e debezium.public.order_items. A sua adopção justifica-se por não requerer qualquer alteração ao schema ou ao código da base de dados fonte — a captura é transparente para o simulador — e por preservar o histórico completo de mutações, incluindo os noop updates que permitem ao MERGE INTO da camada Silver actuar sobre duplicados.
 
MinIO
O MinIO é a plataforma de object storage S3-compatível que serve de substrato ao Data Lake, persistindo os ficheiros Parquet da camada Bronze, os checkpoints Spark e os ficheiros de avaliações em texto. A alternativa mais óbvia seria o AWS S3, mas implicaria dependência de serviços cloud, custos de utilização e impossibilidade de execução local — incompatível com um sistema que corre integralmente em Docker Compose. A alternativa local mais simples seria o sistema de ficheiros do próprio sistema operativo, mas o Spark acede ao storage via conector S3A, o PyArrow usa o protocolo S3, e o file watcher usa boto3 — todos assumem uma API S3. Usar o filesystem local obrigaria a adaptações em cada um destes componentes ou à substituição dos conectores. O HDFS seria a alternativa clássica em arquitecturas Hadoop, mas é significativamente mais pesado para correr em Docker e introduz complexidade operacional desproporcional ao volume de dados do sistema. O MinIO resolve o problema com uma instalação Docker simples que expõe exactamente a API S3 que todos os componentes já esperam, sem qualquer adaptação, e com suporte ao particionamento Hive que o Spark usa para eliminação de partições na leitura. 
 
Apache Spark
O Apache Spark é o motor de processamento distribuído utilizado para todas as transformações Silver e Gold. A sua adopção justifica-se pela integração nativa com o Apache Iceberg — via Iceberg Spark Runtime — que permite operações ACID como MERGE INTO e append transaccional directamente sobre o Data Lake; pela capacidade de Structured Streaming com trigger(availableNow=True), que processa todos os ficheiros Bronze pendentes num único run e termina, adequado ao modelo batch horário do sistema; e pela escalabilidade horizontal, que permite aumentar o volume de dados processado por run sem alteração do código dos jobs.
A alternativa mais directa seria Python com Pandas. No entanto, Pandas não tem equivalente ao MERGE INTO: para deduplicar contra a tabela Silver existente, seria necessário ler a tabela inteira para memória, concatenar com os dados novos, deduplicar e reescrever tudo — uma operação que não escala à medida que a Silver cresce e que, a meio de uma reescrita de ficheiros Parquet no MinIO, não oferece qualquer garantia de atomicidade. Acresce que o rastreamento de quais ficheiros Bronze já foram processados teria de ser implementado de raiz, replicando manualmente o que o sistema de checkpoints do Spark oferece nativamente. O Spark resolve os três problemas — upsert eficiente, atomicidade das escritas e rastreamento de fonte — com mecanismos integrados e testados em produção. 
 
Apache Iceberg
O Apache Iceberg é o formato de tabela open-source utilizado nas camadas Silver e Gold, adotado não por conveniência, mas por responder a três requisitos fundamentais da arquitetura.
O primeiro requisito é a gestão robusta de falhas em produção. Aqui, o sistema combina checkpoints do Spark com o Iceberg de forma complementar. Os checkpoints garantem que os ficheiros do Bronze não são reprocessados em caso de retry do Airflow, enquanto o Iceberg assegura que escritas incompletas (como MERGE INTO ou append interrompidos) não ficam visíveis na tabela. Esta combinação garante consistência end-to-end: a fonte é consumida exatamente uma vez e o destino só é atualizado se a operação for concluída na totalidade. Problemas como processos órfãos causados por SIGKILL obrigaram ainda à implementação de mecanismos de limpeza de checkpoints antes de novas execuções.
O segundo requisito é o suporte a operações de upsert sobre dados CDC. O Debezium gera eventos duplicados por natureza (snapshot + inserts/updates), o que invalida abordagens simples de append. O Iceberg permite resolver este problema com MERGE INTO atómico, permitindo atualizar ou inserir registos com base em chaves como order_id ou review_id, sem necessidade de reprocessar tabelas completas ou introduzir lógica manual de deduplicação.
O terceiro requisito é a eficiência de leitura na camada analítica. As tabelas Gold são particionadas por data e o Iceberg mantém metadados que apontam diretamente para os ficheiros físicos relevantes. Isto permite ao Trino fazer pruning de partições e ler apenas dados relevantes em queries temporais, reduzindo drasticamente o custo de leitura face a uma varredura completa de ficheiros Parquet.
Foram consideradas alternativas como Delta Lake e Apache Hudi, que também suportam ACID e MERGE INTO. No entanto, o Delta apresenta maior acoplamento ao ecossistema Databricks e menor maturidade de integração com Trino, enquanto o Hudi adiciona complexidade operacional (Copy-on-Write vs Merge-on-Read) sem benefício claro para o volume do sistema. O Iceberg destacou-se por oferecer integração sólida e simultânea com Spark e Trino, alinhando-se melhor com os requisitos multi-engine da arquitetura.

Apache Trino
Numa versão anterior da arquitectura, a camada analítica assentava num esquema clássico de data warehouse: os dados das tabelas de factos e dimensões eram materializados no PostgreSQL através de um passo de sincronização dedicado, e o dashboard consultava directamente essa base de dados relacional. Esta abordagem introduzia um problema estrutural: qualquer desfasamento entre o pipeline e a sincronização para o PostgreSQL criava inconsistências na camada de apresentação, e o passo de materialização constituía uma fonte de falha autónoma. Quando a equipa decidiu consolidar toda a camada analítica no lakehouse — eliminando o PostgreSQL como destino intermédio e passando a escrever directamente em tabelas Iceberg sobre MinIO — o problema foi resolvido, mas um novo emergiu: como servir queries interactivas ao dashboard a partir de ficheiros Parquet em object storage?
O Apache Trino resolve este problema de forma directa. Como motor de consulta SQL distribuído com suporte nativo ao formato Iceberg, o Trino acede às tabelas Gold através do mesmo Hive Metastore que o Spark utiliza para escrita. Esta arquitectura de catálogo partilhado elimina qualquer passo de sincronização: assim que o Spark conclui a escrita de uma tabela Gold, o Trino vê imediatamente os novos dados sem qualquer intervenção adicional. Os ficheiros Parquet nunca saem do MinIO — o Trino lê-os directamente, respeitando o esquema Iceberg e o particionamento físico das tabelas, o que reduz o I/O em queries com filtros temporais.
A alternativa natural seria reutilizar o Spark como motor de consulta do dashboard. No entanto, o Spark foi concebido como motor de transformação batch, não como query engine interactivo: o tempo de inicialização de um job Spark situa-se entre 15 e 30 segundos, o que tornaria cada refresh do dashboard perceptivelmente lento e degradaria a experiência de utilização. O Trino, por contraposição, executa queries de agregação sobre vistas pré-calculadas em segundos, sem overhead de inicialização — adequado ao modelo de actualização de cinco em cinco minutos da aplicação Dash.
Para além da performance de leitura, o Trino cumpre um segundo papel arquitectural: servir de camada de abstracção entre o pipeline e o dashboard através de vistas SQL. As seis vistas analíticas — vw_executive, vw_sales_performance, vw_funnel, vw_reviews, vw_trends e vw_category_trends — encapsulam toda a lógica de joins, agregações e cálculos derivados sobre o modelo dimensional. O dashboard nunca acede directamente às tabelas de factos ou dimensões; consome exclusivamente as vistas. Esta separação tem uma consequência prática relevante: alterações no modelo dimensional — mudanças de nomes de colunas, adição de métricas, reformulação de joins — são absorvidas pelas vistas sem qualquer impacto no código do dashboard. As mesmas vistas podem ainda ser consumidas por outras ferramentas de Business Intelligence externas que suportem JDBC ou a API Trino, assegurando a interoperabilidade da camada analítica.

 Hive Metastore
O Hive Metastore é o serviço de catálogo que regista o schema, a localização física e os metadados de particionamento de todas as tabelas Iceberg do sistema. Para cada tabela Silver e Gold, o HMS mantém o mapeamento entre o nome lógico da tabela, o esquema de colunas e tipos, e o caminho no MinIO onde os ficheiros Parquet e os ficheiros de metadados Iceberg residem. É este registo que permite ao Spark e ao Trino referirem-se a lake.silver.orders ou lake.gold.fact_sales como entidades com nome, em vez de caminhos físicos hardcoded no object storage.
A adopção do Hive Metastore não foi uma imposição técnica — o Iceberg suporta catálogos alternativos, nomeadamente o REST Catalog nativo, o JDBC Catalog e o Nessie. A escolha recaiu sobre o HMS por uma razão operacional concreta: é o único catálogo com integração de primeira classe, estável e amplamente documentada, em simultâneo no Spark e no Trino. O REST Catalog do Iceberg, apesar de ser a opção mais moderna e arquitecturalmente mais limpa, apresentava suporte menos maduro nos dois motores à data de implementação, com configurações menos estáveis e documentação mais escassa. Utilizar o HMS como catálogo partilhado era a opção que minimizava o risco operacional mantendo a compatibilidade total entre os dois motores.
A consequência prática desta escolha é o catálogo único partilhado: quando o Spark escreve uma tabela Gold e regista o commit no HMS, o Trino vê imediatamente essa tabela sem qualquer passo de sincronização adicional. Os dois motores operam sobre o mesmo estado de catálogo, o que garante consistência entre a camada de transformação e a camada de consulta sem intervenção manual.

Apache Airflow
O Apache Airflow é o orquestrador de pipelines que gere a execução das transformações Silver e Gold. A alternativa mais simples seria um cron job que corre um script Python de hora a hora — mas um cron job não tem noção de dependências entre tasks. Para replicar o grafo de dependências do pipeline, o script teria de encadear manualmente chamadas com verificação de estado, gerir paralelismo com threads ou subprocessos, e implementar lógica de retry dentro do próprio código — replicando, de forma mais frágil, o que o Airflow oferece nativamente.
A adopção do Airflow justifica-se por quatro propriedades. Primeiro, a definição declarativa de dependências em DAG: o grafo determina que as tasks Silver correm em paralelo, que as tasks Gold de factos só arrancam após gold_dimensions completar, e que init_views aguarda a conclusão de todas as tasks Gold — sem qualquer lógica de sincronização manual. Segundo, retentativas automáticas por task com atraso configurável: o pipeline teve falhas reais de jobs Spark por timeout e por JVMs órfãos; sem retentativas automáticas, cada falha exigiria intervenção manual para re-executar apenas a task afectada sem repetir as que já concluíram com sucesso. Terceiro, auditabilidade: o Airflow mantém o registo de cada run — quais tasks correram, quando, quanto tempo demoraram e o log de output — o que permite diagnosticar gaps de dados sem aceder directamente às tabelas. Quarto, paralelismo declarativo: tasks independentes correm em simultâneo sem threading manual no código.
O sistema utiliza duas DAGs independentes com cadências distintas. A DAG trendmart_pipeline corre de hora a hora. 
Em caso de falha, o Airflow retenta as tasks Silver após 15 minutos e as tasks Gold após 5 minutos. A DAG trendmart_ml_pipeline corre diariamente às 03:00 UTC, após o pipeline Gold ter os dados do dia completos, e executa a task demand_forecast, que treina o modelo de previsão de procura e escreve os resultados em lake.gold.ml_demand_forecast. Em caso de falha, retenta após 10 minutos.




4. Exploração e Análise de Dados

4.1 Organização Geral do Sistema de Dashboarding
A camada de visualização do TrendMart é implementada como uma aplicação web em Plotly Dash com Bootstrap, estruturada em seis dashboards temáticos. Cada dashboard foi desenhado para responder às necessidades analíticas dos perfis de utilizador definidos no capítulo 2.
A aplicação está disponível em http://localhost:8050 e atualiza automaticamente os dados a cada cinco minutos através do componente dcc.Interval, garantindo que a informação apresentada reflecte o estado mais recente da camada Gold sem intervenção manual.

4.1.1 Arquitetura da Camada de Visualização
A aplicação está organizada em dois módulos Python com responsabilidades claramente separadas.
O módulo data.py implementa a camada de acesso a dados. Este módulo contém exclusivamente funções responsáveis por construir queries SQL parametrizadas, executá-las sobre as vistas do Trino através do cliente trino.dbapi, e devolver os resultados em DataFrames Pandas.
Cada função corresponde a um dashboard específico — get_executive, get_sales_performance, get_funnel, get_reviews, get_trends, get_category_trends e get_demand_forecast — recebendo como parâmetro a janela temporal de análise em dias. Esta separação permite isolar a lógica de acesso a dados, facilitando testes, manutenção e possível substituição da fonte de dados sem impacto na camada de apresentação.
O módulo app.py implementa a camada de apresentação. É responsável pela definição do layout da aplicação, organizado em abas (dcc.Tabs) correspondentes aos seis dashboards. Este módulo regista o callback principal da aplicação, que reage tanto à alteração de dashboard como ao evento de atualização periódica, e delega a renderização de cada vista para funções específicas (_tab_executive, _tab_sales, _tab_funnel, _tab_reviews, _tab_trends, _tab_ml).
A aplicação utiliza uma arquitetura de callback único, na qual um único @app.callback re-renderiza integralmente o dashboard ativo a cada atualização. Esta abordagem simplifica a gestão de estado, reduz a complexidade de sincronização entre componentes e garante consistência dos dados apresentados em cada refresh.
A utilização de vistas Trino como interface entre o pipeline analítico e a camada de visualização apresenta três vantagens principais. Em primeiro lugar, os dashboards recebem dados já agregados e prontos a consumir, reduzindo significativamente a complexidade de processamento em Python. Em segundo lugar, alterações no modelo dimensional são encapsuladas nas vistas, sem impacto direto na camada de visualização. Em terceiro lugar, as mesmas vistas podem ser reutilizadas por outras ferramentas de Business Intelligence, assegurando interoperabilidade da camada analítica.

4.2 Dashboards Implementados

4.2.1 Dashboard Executivo
O Dashboard Executivo é o ponto de entrada do sistema e serve o perfil do Gestor Executivo. Apresenta uma visão agregada do desempenho global da plataforma com histórico completo, combinando cartões KPI de topo com gráficos de barras temporais e uma tabela de detalhe. O objetivo é permitir ao gestor responder de forma imediata a questões sobre a saúde geral do negócio — quanto se vendeu, quantas encomendas foram processadas e qual o valor médio por transação — sem necessidade de navegar por outros dashboards.

Métricas e Visualizações

Cartões KPI — RF1.1
Os quatro cartões de topo apresentam as métricas agregadas de todo o histórico disponível: Receita Total (em euros), Total de Pedidos, Valor Médio por Pedido e Clientes Únicos (ver Figura X). O eixo vertical usa formatação monetária (€) e o tooltip apresenta o valor exato ao passar o cursor sobre cada barra.

5. O Sistema de Análise de Tendências de Aquisição de Produtos

5.1 Definição do problema e compreensão dos elementos de análise envolvidos
O problema central abordado neste sistema é a previsão da procura de produtos por categoria ao longo de um horizonte temporal de sete dias. Em contextos de comércio eletrónico, a antecipação da procura é determinante para a gestão eficiente de inventário, para o planeamento logístico e para a ativação proativa de alertas a vendedores. A ausência de previsões fiáveis conduz frequentemente a rupturas de stock, excesso de aprovisionamento ou falhas na capacidade de resposta ao cliente. O sistema Projeto responde a este problema através de um pipeline de machine learning integrado no ecossistema de dados da plataforma. A análise de tendências de aquisição é tratada como um problema de previsão de séries temporais por categoria de produto, em que o objetivo é estimar o número de encomendas esperadas para cada categoria nos próximos sete dias, com base em padrões históricos de consumo. A solução é construída sobre os dados acumulados nas camadas Gold do data lake, nomeadamente fact_sales, dim_category e dim_date, e é executada diariamente às 03:00 UTC através de um DAG Apache Airflow dedicado (trendmart_ml_pipeline). Os resultados das previsões são disponibilizados no dashboard analítico da plataforma, no separador ML Insights, e persistidos na tabela lake.gold.ml_demand_forecast após validação de qualidade.

Processo de preparação e engenharia de features
A preparação dos dados é realizada no módulo machine_learning/features/timeseries_features.py, em Apache Spark, seguindo as etapas descritas abaixo:
Agregação diária por categoria
Os dados de vendas são agrupados por category_id e purchase_date, calculando o número de encomendas distintas (orders_count) e a receita total (total_revenue) para cada par categoria-dia. Esta operação transforma dados ao nível de encomenda num formato de série temporal diária.
Construção de features de lag (autocorrelação temporal)
Para capturar padrões de dependência temporal, são criadas quatro variáveis de desfasamento temporal. A variável lag_1 representa o número de encomendas no dia anterior (t−1), servindo como indicador imediato da tendência recente. A variável lag_7 regista as encomendas de sete dias antes (t−7), permitindo ao modelo capturar sazonalidade semanal  por exemplo, padrões de compra distintos entre fins de semana e dias úteis que se repetem ciclicamente. De forma análoga, lag_14 incorpora o valor de catorze dias antes (t−14), capturando padrões bi-semanais que o lag semanal isolado não consegue representar. Por fim, rolling_7d_mean calcula a média móvel dos últimos sete dias, introduzindo uma variável de tendência suavizada que atenua variações abruptas e pontuais no histórico de encomendas.
Features de calendário
A partir de dim_date, são incorporadas variáveis que descrevem o contexto temporal de cada observação: dia da semana (day_of_week), indicador de fim de semana (is_weekend), mês (month) e trimestre (quarter). Estas features permitem ao modelo distinguir padrões sazonais recorrentes.
Filtragem de registos incompletos
Após a construção das features de lag, são removidas as linhas em que lag_14 é nulo — o que corresponde aos primeiros 14 dias de histórico por categoria, nos quais não existe informação suficiente para calcular todas as variáveis de desfasamento. Adicionalmente, categorias com menos de 20 dias de histórico válido são excluídas do treino, assegurando robustez mínima dos modelos.
Vetor de features final
O conjunto de features que alimenta o modelo é composto por 8 variáveis: ["lag_1", "lag_7", "lag_14", "rolling_7d_mean", "day_of_week", "is_weekend", "month", "quarter"]. A variável alvo (label) é orders_count, o número de encomendas registadas no dia corrente.

5.3 Identificação e fundamentação da técnica de análise
Técnica selecionada: Regressão Linear com features de lag (por categoria)
A técnica de análise adotada é a regressão linear (LinearRegression da biblioteca Spark MLlib), aplicada de forma independente a cada categoria de produto. Esta abordagem designada por modelos per-category permite capturar dinâmicas de procura específicas de cada categoria, que seriam diluídas num modelo único global.
Justificação da escolha
A regressão linear foi selecionada em detrimento de outras alternativas (e.g., ARIMA, redes neuronais recorrentes, XGBoost) com base nos seguintes critérios:
Compatibilidade com o ecossistema tecnológico: O pipeline de dados da plataforma assenta integralmente em Apache Spark. A utilização de Spark MLlib assegura que o treino, avaliação inferência ocorrem no mesmo ambiente de processamento distribuído, sem dependências externas ou conversões de formato.
Interpretabilidade: A regressão linear produz coeficientes diretamente interpretáveis, permitindo compreender a contribuição de cada feature (e.g., o peso do lag semanal vs. o dia da semana) na previsão. Isto é valioso num contexto académico e de auditoria.
Representação implícita de séries temporais através de features de lag: Em vez de utilizar modelos de séries temporais clássicos (ARIMA, Prophet), que requerem bibliotecas externas e são mais complexos de paralelizar por categoria em Spark, a abordagem adotada transforma o problema de previsão numa tarefa de regressão supervisionada. Os padrões temporais são codificados explicitamente nas features de lag e de médias móveis, tornando o problema tratável por qualquer algoritmo de regressão.
Eficiência computacional: Com um número de categorias da ordem das dezenas (dataset Olist), o treino independente de modelos de regressão linear é extremamente rápido, adequado ao contexto de execução diária com restrições de memória (driver.memory=1g, local[2]).
Regularização L2: O parâmetro regParam=0.1 introduz regularização Ridge, penalizando coeficientes excessivamente elevados e reduzindo o risco de overfitting em categorias com poucos dados históricos.

5.4 Construção do modelo de análise




O modelo é implementado no módulo machine_learning/models/demand_forecast.py, seguindo uma arquitetura de treino por categoria. Para cada categoria com histórico suficiente (≥ 20 dias), é executado o seguinte processo:

Divisão treino/teste cronológica
Os dados são ordenados por data e divididos na proporção 80/20, preservando a ordem temporal. Esta escolha é fundamental: uma divisão aleatória introduziria data leakage, uma vez que dados futuros seriam usados para treinar o modelo, resultando numa avaliação irrealisticamente otimista.

Construção do pipeline Spark MLlib
O pipeline é composto por dois estágios encadeados, VectorAssembler (agrega as 8 features num vetor denso compatível com os algoritmos MLlib) LinearRegression (modelo de regressão com maxIter=100 e regParam=0.1)

Treino e registo no MLflow
Cada modelo treinado é registado no MLflow com o ID de experiência demand_forecasting, incluindo os parâmetros (category, algorithm, reg_param), as métricas de avaliação (rmse, mae) e o artefacto completo do pipeline Spark serializado em MinIO.

Geração de previsões para os próximos 7 dias
Para cada categoria, são construídas 7 linhas de previsão (D+1 a D+7), com as respetivas features de lag calculadas a partir dos últimos valores históricos conhecidos, e as features de calendário derivadas diretamente das datas futuras. 

A primeira tarefa executa o treino, avaliação e escrita na sandbox, passando as métricas agregadas via XCom. A segunda tarefa valida essas métricas contra os limiares configurados e, em caso de aprovação, promove os dados da sandbox para a tabela Gold de produção.





5.5 Validação do desempenho do modelo




O desempenho de cada modelo por categoria é avaliado no conjunto de teste (os 20% mais recentes de cada série temporal) através de duas métricas complementares: O RMSE penaliza de forma quadrática os erros de maior magnitude, tornando-o sensível a previsões com desvios elevados em dias de pico ou de rutura atípica. É a métrica primária de controlo de qualidade. O MAE representa o erro médio absoluto em unidades de encomendas, sendo mais robusto a outliers e de interpretação direta: um MAE de 5 significa que, em média, o modelo erra por 5 encomendas por dia. O sistema implementa um padrão de promoção condicional (Analytical Sandbox): os modelos treinados são primeiro escritos numa zona de sandbox (lake.sandbox) e só são promovidos para produção (lake.gold) se as métricas médias, calculadas sobre todas as categorias, cumprirem simultaneamente os limiares configurados: RMSE médio ≤ 50.0 encomenda, MAE médio≤ 40.0 encomenda
Se qualquer um dos limiares for excedido, a tarefa promote_to_gold falha com um ValueError, o Airflow marca a execução como falhada, e a tabela Gold retém os dados da execução anterior  garantindo que previsões de qualidade inferior nunca chegam ao dashboard de produção.
Rastreabilidade com MLflow
Toda a informação de treino e avaliação é registada no MLflow, acessível em http://localhost:5001, na experiência demand_forecasting. Por cada execução do DAG, existe um run MLflow por categoria, contendo parâmetros, métricas e o artefacto completo do modelo. Esta rastreabilidade permite comparações históricas de desempenho e auditoria do comportamento dos modelos ao longo do tempo.

 Resumo
O TrendMart é um sistema de análise de dados de grande escala desenvolvido para suporte à decisão num contexto de e-commerce, tendo como base o dataset público Olist, que contém dados reais de transações, avaliações e comportamento de navegação do mercado brasileiro.
O sistema foi concebido para servir seis perfis de analistas de dados — Gestor Executivo, Analista de Vendas, Analista de Marketing, Gestor de Customer Experience, Analista de Tendências e Data Scientist — cada um com necessidades analíticas distintas, num total de 22 requisitos funcionais e 6 requisitos não funcionais levantados por análise de cenário.
A arquitetura adota um Data Lake com organização em três camadas: Bronze, para preservação imutável dos dados originais; Silver, para limpeza e validação; e Gold, para estruturas analíticas otimizadas para consulta. Os dados são gerados por um simulador de eventos, ingeridos via Apache Kafka com Change Data Capture por Debezium, processados em Apache Spark, persistidos em formato Apache Iceberg e expostos através de vistas Trino. O processo de integração é orquestrado pelo Apache Airflow com cadência horária.
A camada analítica disponibiliza seis dashboards interativos desenvolvidos em Plotly Dash, um por perfil, com atualização automática a cada cinco minutos. O sistema inclui ainda um pipeline de machine learning diário que treina um modelo de previsão de procura por categoria de produto, disponibilizando previsões para os sete dias seguintes com métricas de desempenho associadas.
A reprodutibilidade total do sistema é garantida por Docker Compose, sem dependências externas ao código-fonte e ao dataset Olist.

palavras-chave: Medallion Architecture, Hive metastore, Change Data Capture, Trino, Apache Iceberg, Data Lakehouse

1. Definição do Sistema

1.1 Contexto de Aplicação
O Projeto baseia-se na simulação de uma plataforma de e-commerce de dimensão real, contendo informação sobre vendas, clientes, produtos, regiões geográficas e avaliações de encomendas. O contexto operacional do sistema simula uma organização de retalho online com presença nacional no Brasil, com dezenas de milhares de produtos distribuídos por mais de 70 categorias distintas, e com operações cobrindo os 27 estados brasileiros. Esta organização lida diariamente com volumes elevados de dados provenientes de fontes heterogéneas e com naturezas distintas: dados transacionais de encomendas registados em base de dados relacional, eventos comportamentais de sessões de utilizador em tempo real (clickstream), e avaliações textuais submetidas pelos clientes após a entrega dos produtos.
A complexidade analítica deste cenário resulta da combinação de três fatores. Primeiro, a heterogeneidade das fontes de dados: dados estruturados de vendas, dados semi-estruturados de eventos de navegação e dados não estruturados de texto de avaliações, cada um com volume, velocidade e natureza técnica distintos. Segundo, o requisito de análise temporal contínua: a detecção de tendências emergentes exige que os dados estejam disponíveis para análise de forma regular e atualizada, sem depender de extrações manuais ou processos batch de execução rara. Terceiro, a necessidade de suporte à decisão multidimensional: necessidades de análise distintas e complementares que o sistema deve satisfazer de forma coerente.
Para responder a estes desafios, o implementa uma arquitetura de Data Lakehouse com padrão Medallion (Bronze → Silver → Gold), integrando tecnologias de processamento distribuído, ingestão em tempo real, orquestração de pipelines e aprendizagem automática, num ambiente completamente containerizado e reproduzível. O sistema gera continuamente dados simulados com base no dataset Olist e processa-os através de um pipeline automático que os transforma progressivamente em informação analítica de valor.

1.2 Motivação e Objetivos do Trabalho
As organizações de retalho online que operam com análises baseadas em extrações manuais com latência de dias ou semanas, impede a detecção atempada de tendências. Este projeto substitui este modelo por um pipeline contínuo e automatizado que transforma dados heterogêneos em suporte à decisão para múltiplos perfis.
Integrar três fontes heterogêneas num pipeline unificado e tolerante a falhas. Estruturar os dados em camadas de qualidade crescente (Bronze/Silver/Gold) com rastreabilidade completa. Calcular métricas quantitativas de tendência. Produzir previsões diárias de procura por categoria. Expor os resultados num dashboard interativo com atualização automática de 5 em 5 minutos.

1.3 Análise da Viabilidade do Processo
Viabilidade técnica: Todas as ferramentas adotadas são de código aberto, maduras e amplamente utilizadas em produção: Apache Spark, Kafka, Iceberg, Trino, Airflow e MLflow têm adoção documentada em empresas como Netflix, LinkedIn e Uber. Nenhuma integração envolve tecnologia experimental.

Viabilidade operacional: O sistema é completamente reprodutível a partir do código-fonte. Um único comando (python Extra/start.py) inicializa os doze serviços Docker, cria a estrutura de dados e inicia o pipeline. O processo completo de arranque demora menos de 15 minutos numa máquina com 16 GB de RAM.
Viabilidade económica: O custo de operação do sistema é praticamente nulo. Todo o software utilizado é open-source e gratuito. A infraestrutura corre integralmente numa única máquina de desenvolvimento desktop de uso corrente serve como ambiente completo de produção.
Viabilidade de dados: A organização dispõe de dados operacionais contínuos provenientes das três fontes identificadas, registros transacionais de compras, eventos de sessão dos utilizadores e avaliações de clientes acumulados ao longo de vários anos de operação. O volume histórico disponível é suficiente para suportar todos os requisitos analíticos.

1.4 Recursos e Equipa de Trabalho
Recursos Humanos: O projeto foi desenvolvido por uma equipa de quatro estudantes de Engenharia e Ciência de Dados. Todos os membros participaram activamente em todas as fases do projecto desde a configuração da infraestrutura até ao desenvolvimento do dashboard adaptando o seu envolvimento às necessidades de cada etapa. Esta abordagem garantiu que toda a equipa adquiriu conhecimento transversal do sistema e que nenhuma componente ficou dependente de um único elemento. 
Recursos Materiais: O sistema corre integralmente em Docker Compose, numa única máquina de desenvolvimento. Os requisitos mínimos são 16GB de RAM, 4 núcleos de CPU e 20 GB de armazenamento SSD. Todo o software é open-source e gratuito.

1.5 Plano de Execução do Projeto
O desenvolvimento seguiu uma abordagem iterativa orientada a camadas, na ordem natural do fluxo de dados da infraestrutura.

O plano de execução do projeto está organizado em nove fases sequenciais e parcialmente sobrepostas. A primeira fase corresponde ao levantamento e análise de requisitos, bem como à definição da arquitetura global do sistema. Segue-se a configuração do ambiente de infraestrutura, assegurando que todos os serviços e dependências técnicas estejam operacionais antes do início do desenvolvimento. A terceira fase contempla a implementação das fontes de dados e dos mecanismos de geração e publicação de eventos. Com as fontes activas, procede-se à construção da camada de ingestão, responsável por recolher e persistir os dados em bruto. A quinta fase compreende a transformação e limpeza dos dados ingeridos, produzindo uma camada estruturada e validada. Segue-se o desenvolvimento do modelo analítico dimensional e a exposição de vistas de consulta para consumo pelos utilizadores. As fases sete e oito decorrem em paralelo: o desenvolvimento dos modelos de aprendizagem automática e a construção da interface de visualização e análise. Por fim, a nona fase abrange a redação do relatório e a documentação técnica do sistema.

2. Levantamento e Análise de Requisitos

O desenvolvimento de um sistema de análise de dados de grande escala exige, antes de qualquer decisão técnica, uma compreensão clara de quem vai usar o sistema e para quê. Sem esta ancoragem, corre-se o risco de construir uma infraestrutura tecnicamente sofisticada mas incapaz de responder às perguntas de negócio que justificam a sua existência. O levantamento de requisitos é, por isso, o primeiro passo estruturante do projeto no momento em que se traduzem necessidades de negócio em especificações concretas que guiam todas as decisões de arquitetura e implementação subsequentes.

O sistema serve uma organização de e-commerce com múltiplas áreas funcionais, cada uma com perspectivas, objetivos e ritmos de trabalho distintos. Em empresas reais, estas necessidades analíticas estão tipicamente distribuídas entre dois tipos de sistemas de informação: sistemas que gerem dados operacionais e transacionais como vendas, receita e inventário, e sistemas que centralizam dados de comportamento de cliente, interações, feedback e satisfação. O Projeto replica analiticamente os dados que estes dois tipos de sistemas geram, consolidando-os numa camada analítica unificada.

A interface que concretiza este serviço é o dashboard desenvolvido pela equipa `data_analytics`. Esta equipa é técnica e única mas o produto que entrega ramifica-se em seis abas, cada uma orientada a um perfil de utilizador distinto. 
Os perfis de natureza ERP: Gestor Executivo, Analista de Vendas e Analista de Tendências  consomem dados transacionais e operacionais: receita, volume de vendas, crescimento e anomalias. 
Os perfis de natureza CRM: Analista de Marketing, Gestor de Customer Experience e Data Scientist consomem dados comportamentais e relacionais: funil de conversão, sentimento de cliente e previsão de procura. É esta ramificação que serve de estrutura ao levantamento de requisitos deste capítulo: cada aba corresponde a um perfil, cada perfil tem perguntas de negócio próprias, e cada pergunta origina requisitos funcionais concretos que o sistema tem de satisfazer.

Este capítulo organiza-se em três secções: o método de levantamento adotado (2.1), a caracterização dos seis perfis com os respetivos requisitos funcionais e não funcionais (2.2), e a validação de cada requisito contra a implementação real do sistema (2.3).

2.1 Método Adotado
O método adotado foi a análise de cenário orientada a perfis de utilizador. Para cada um dos seis perfis identificados, foram definidas as perguntas de negócio que o sistema deve ser capaz de responder, os dados necessários para o efeito, e as restrições operacionais específicas ao seu contexto de uso. Os requisitos funcionais derivam diretamente dessas perguntas, um requisito está satisfeito se e só se a pergunta que lhe corresponde pode ser respondida com o sistema construído.
Esta abordagem tem três vantagens metodológicas. Primeiro, força a ancoragem dos requisitos em necessidades concretas de negócio em vez de capacidades técnicas abstratas. Segundo, facilita a priorização  funcionalidades partilhadas por múltiplos perfis têm maior prioridade do que as específicas a um único perfil, o que guiou as decisões de arquitetura no sentido de maximizar o valor entregue com os recursos disponíveis. Terceiro, torna a validação dos requisitos sistemática  na secção 2.3, cada requisito é verificado contra a implementação real do sistema, tornando explícita a correspondência entre o que foi especificado e o que foi construído.
Para além dos requisitos funcionais por perfil, foram identificados requisitos não funcionais transversais relativos a latência, independência, rastreabilidade, escalabilidade e reprodutibilidade que condicionam a arquitetura e não são específicos a nenhum perfil de utilizador.

2.2 Organização dos Requisitos Levantados

2.2.1	Perfil Gestor Executivo
O Gestor Executivo é responsável por monitorizar a saúde global do negócio e tomar decisões estratégicas com base em indicadores de desempenho de alto nível. O seu interesse não está no detalhe operacional — está na evolução da receita, no volume de encomendas e no crescimento período a período. Trabalha com horizontes temporais de 90 dias e necessita que os dados estejam atualizados com frequência suficiente para detetar tendências antes que se tornem problemas.

Perguntas de negócio
Qual foi a receita total, o número de encomendas, os clientes únicos e o valor médio por encomenda no período?
Como evoluiu a receita e o volume de encomendas diariamente ao longo dos últimos 90 dias?

Requisitos funcionais derivados
RF1.1 — KPIs globais diários: receita total, volume de encomendas, clientes únicos e valor médio por encomenda, em série temporal de 90 dias.
RF1.2 — Evolução diária da receita e do volume de encomendas em formato de série temporal.

2.2.2	Perfil Analista de Vendas
O Analista de Vendas é responsável por avaliar o desempenho comercial da plataforma ao nível do detalhe — por categoria de produto e por região geográfica. O seu interesse está em perceber quais as categorias que mais contribuem para a receita, como se distribui geograficamente a base de clientes, e qual a composição da receita entre produto e frete. Trabalha com horizontes de 90 dias e necessita de vistas desagregadas que permitam identificar oportunidades e áreas de melhoria no portfólio de produtos. Este perfil é de natureza ERP — consome dados transacionais com granularidade por categoria, região e estado.

Perguntas de negócio
Quais são as categorias de produto com maior contribuição para a receita?
Como se distribui a receita por região e por estado?
Qual a decomposição da receita entre valor de produto e custos de frete por categoria?

Requisitos funcionais derivados
RF2.1 — O sistema deve disponibilizar as top 10 categorias por receita total num horizonte de 90 dias.
RF2.2 — O sistema deve disponibilizar a distribuição de receita por macrorregião e por estado.
RF2.3 — O sistema deve disponibilizar a decomposição de receita em valor de produto e valor de frete por categoria e por período.

2.2.3	Perfil Analista de Marketing
O Analista de Marketing é responsável por monitorizar o comportamento de navegação dos utilizadores na plataforma e avaliar a eficácia do funil de conversão. O seu interesse está em perceber em que etapa do funil os utilizadores abandonam, como se distribui o comportamento por tipo de dispositivo, como evoluem as sessões e utilizadores ativos ao longo do tempo, e se o comportamento difere entre dias úteis e fins de semana. Trabalha com horizontes de 7 dias para análise do funil e 30 dias para análise de padrões semanais. Este perfil é de natureza CRM — consome dados comportamentais gerados pelo clickstream.

Perguntas de negócio
Qual é o volume de eventos em cada etapa do funil de conversão e onde ocorre maior abandono?
Como se distribui o comportamento de navegação por dispositivo — mobile, desktop e tablet?
Como evoluiu o número de sessões e utilizadores ativos nos últimos 7 dias?
O comportamento de navegação difere entre dias úteis e fins de semana?

Requisitos funcionais derivados
RF3.1 — O sistema deve disponibilizar o funil completo de eventos — desde session_start até order_placed — com contagens por etapa.
RF3.2 — O sistema deve desagregar os eventos por tipo de dispositivo de acesso.
RF3.3 — O sistema deve disponibilizar a evolução temporal dos top 5 eventos mais frequentes com granularidade diária numa janela de 7 dias.
RF3.4 — O sistema deve disponibilizar o número de sessões e utilizadores únicos por dia.
RF3.5 — O sistema deve comparar o volume de eventos e sessões entre dias úteis e fins de semana numa janela de 30 dias.

2.2.4   Perfil Gestor de Customer Experience
O Gestor de Customer Experience é responsável por monitorizar a qualidade da experiência do cliente após a compra, nomeadamente através da análise das avaliações e do sentimento expresso. O seu interesse está em identificar padrões de insatisfação — categorias com ratings baixos, regiões com avaliações negativas recorrentes, deterioração temporal do sentimento — antes que se traduzam em perda de clientes. Trabalha com horizontes de 90 dias e necessita de vistas que permitam detetar precocemente sinais de degradação da experiência. Este perfil é de natureza CRM — consome dados de feedback e satisfação de cliente.

Perguntas de negócio
Como se distribui o sentimento das avaliações entre positivo, neutro e negativo?
Como evoluiu o rating médio ao longo do tempo?
Quais as categorias com maior volume de reviews e qual o seu rating médio?
Como se distribui o sentimento por região geográfica?

Requisitos funcionais derivados
RF4.1 — O sistema deve disponibilizar a distribuição de reviews por sentimento — positivo, neutro e negativo — com base na classificação do rating.
RF4.2 — O sistema deve disponibilizar a evolução do rating médio diário numa janela de 90 dias.
RF4.3 — O sistema deve disponibilizar as top 10 categorias por volume de reviews com o respetivo rating médio.
RF4.4 — O sistema deve disponibilizar a distribuição de sentimento por macrorregião geográfica.

2.2.5 Perfil Analista de Tendências
O Analista de Tendências é responsável por identificar padrões de crescimento e aceleração da procura no comportamento das vendas. Ao contrário do Analista de Vendas, que olha para o desempenho absoluto, este perfil foca-se na variação relativa — crescimento semana-sobre-semana e aceleração dessa taxa de crescimento. É o perfil mais diretamente alinhado com o objetivo central do enunciado: a deteção precoce de tendências de produto. Trabalha com horizontes de 60 dias e necessita de métricas de variação temporal com granularidade semanal. Este perfil é de natureza ERP — consome dados transacionais agregados com métricas derivadas de séries temporais.

Perguntas de negócio
Qual foi o crescimento da receita e do volume de encomendas semana-sobre-semana?
A taxa de crescimento está a acelerar ou a desacelerar?
Quais as categorias com maior e menor crescimento médio nos últimos 30 dias?

Requisitos funcionais derivados
RF5.1 — O sistema deve calcular e disponibilizar o crescimento semana-sobre-semana da receita e do volume de encomendas.
RF5.2 — O sistema deve calcular e disponibilizar a aceleração da procura — variação da taxa de crescimento entre semanas consecutivas.
RF5.3 — O sistema deve disponibilizar o crescimento médio por categoria numa janela de 30 dias.

2.2.6 Perfil Data Scientist
O Data Scientist é responsável por interpretar os resultados dos modelos preditivos treinados pelo pipeline de machine learning e monitorizar a qualidade das previsões ao longo do tempo. O seu interesse está na disponibilidade das previsões de procura por categoria para os próximos sete dias e nas métricas de desempenho do modelo que as suportam. Ao contrário dos restantes perfis, não consome apenas dados de negócio — consome também métricas de avaliação de modelos que lhe permitem aferir a fiabilidade das previsões antes de as utilizar em decisões operacionais. Este perfil é de natureza CRM — as previsões de procura derivam de padrões comportamentais e transacionais dos clientes.

Perguntas de negócio
Qual é a previsão de encomendas para os próximos 7 dias por categoria de produto?
Qual o desempenho preditivo do modelo em termos de RMSE e MAE por categoria?

Requisitos funcionais derivados
RF6.1 — O sistema deve treinar um modelo de previsão de procura por categoria e persistir as previsões para os sete dias seguintes com as métricas de desempenho associadas.
RF6.2 — O sistema deve disponibilizar as previsões de procura e as métricas do modelo num dashboard interativo, atualizadas a cada execução diária do pipeline ML.

2.2.7 Requisitos Não Funcionais
Os requisitos não funcionais identificados são transversais a todos os perfis e condicionam as decisões de arquitetura do sistema.
RNF1 — Latência analítica: os dados disponibilizados no dashboard devem refletir os eventos gerados pelo simulador com uma latência máxima de uma hora, correspondente à cadência do pipeline batch de transformação Silver → Gold. O dashboard atualiza as suas vistas de cinco em cinco minutos.
RNF2 — Idempotência: qualquer operação de processamento ou carga deve poder ser repetida sem duplicar ou corromper dados. Este requisito aplica-se a todos os jobs Spark de transformação Silver (via operação MERGE INTO Iceberg), ao pipeline Gold (via modo append incremental com watermark) e às tabelas ML (via createOrReplace a cada execução diária).
RNF3 — Rastreabilidade: os dados originais devem ser preservados em formato imutável na camada Bronze do Data Lake, independentemente de qualquer transformação posterior. Em caso de erro na camada Silver ou no pipeline Gold, deve ser possível reprocessar os dados originais sem contactar novamente a fonte.
RNF4 — Reprodutibilidade: o sistema deve ser completamente reproduzível a partir do código-fonte e do dataset Olist público, sem dependência de dados externos não disponíveis. A reprodutibilidade é garantida pelo Docker Compose e pelo dataset Olist carregado na inicialização do simulador.
RNF5 — Escalabilidade horizontal: a arquitetura deve suportar aumento do volume de dados sem alteração estrutural do pipeline. A adoção do Apache Spark como motor de processamento garante que os jobs de transformação escalam horizontalmente com a adição de executores, sem alteração do código.
RNF6 — Orquestração auditável: a execução dos pipelines de dados deve ser gerida por um orquestrador com suporte a dependências entre tasks, retentativas automáticas e registo do estado de execução de cada run. Este requisito fundamenta a escolha do Apache Airflow.

3. Implementação do Sistema de Análise

3.1 Apresentação Geral
O TrendMart é um sistema de análise de dados em grande escala orientado ao suporte à decisão no domínio do retalho de e-commerce. O seu principal objectivo é simular, ingerir, transformar e analisar continuamente dados produzidos por uma plataforma digital de comércio electrónico, convertendo eventos operacionais heterogéneos numa infraestrutura analítica integrada capaz de suportar exploração de negócio, monitorização operacional e análise preditiva.
O sistema foi concebido para reproduzir não apenas os dados normalmente gerados por uma plataforma de e-commerce moderna, mas também os próprios padrões arquiteturais utilizados em ecossistemas reais de dados distribuídos. Neste contexto, o pipeline integra mecanismos de streaming de eventos, captura de alterações transacionais (CDC), armazenamento object-based, processamento distribuído com Apache Spark, tabelas analíticas Apache Iceberg, consultas SQL via Trino e orquestração contínua através do Apache Airflow.
A arquitetura implementa quatro capacidades principais. A primeira corresponde à ingestão contínua de dados em tempo quase-real. Eventos de navegação, transações de compra e avaliações de clientes são produzidos continuamente pelo simulador e capturados através de três mecanismos de ingestão distintos, cada um adaptado às características operacionais da respectiva fonte. A segunda capacidade corresponde à preparação e governação de dados. Os dados brutos provenientes das diferentes fontes são progressivamente transformados em representações limpas, tipadas, deduplicadas e semanticamente enriquecidas através de uma arquitetura em múltiplas camadas baseada no padrão Medalhão. A terceira capacidade corresponde à análise descritiva e exploratória. Sobre as tabelas analíticas produzidas pelo pipeline é construído um modelo dimensional em estrela, posteriormente exposto através de vistas SQL optimizadas para consumo por dashboards interativos e ferramentas externas de Business Intelligence. A quarta capacidade corresponde à análise preditiva. Modelos de Machine Learning treinados sobre os dados históricos do sistema produzem previsões de procura por categoria, disponibilizadas como artefactos analíticos integrados no ecossistema de consumo.

3.1.1 Arquitetura Global
A arquitectura organiza-se em camadas funcionais com responsabilidades bem delimitadas e interfaces explícitas entre si.
Resumidamente, o pipeline inicia-se num simulador contínuo de e-commerce que gera três fontes de dados heterogéneas: eventos de clickstream publicados em Kafka, ordens de compra persistidas em PostgreSQL e avaliações armazenadas em MinIO. Três mecanismos de ingestão independentes capturam continuamente estas fontes e persistem os dados em bruto na camada Bronze do Data Lakehouse sob a forma de ficheiros Parquet particionados.
Sobre esta camada executam-se transformações Spark orquestradas pelo Airflow, responsáveis por produzir tabelas Silver limpas, tipadas e deduplicadas, posteriormente convertidas num modelo dimensional Gold orientado para análise de negócio. Em paralelo, uma pipeline de Machine Learning treina modelos preditivos sobre os dados históricos Gold e publica previsões de procura consumidas pelo dashboard analítico.
A camada final do sistema corresponde ao dashboard interactivo, que consulta directamente as vistas analíticas e apresenta os resultados através de múltiplas perspectivas de análise de negócio.
A construção, execução e manutenção do sistema encontram-se distribuídas por cinco equipas funcionais com responsabilidades explicitamente delimitadas: Engenharia de Dados, Engenharia Analítica, Machine Learning, Engenharia de Infraestrutura e Analítica de Dados.

3.1.2 Equipas e Responsabilidades
O pipeline de dados do TrendMart encontra-se organizado segundo uma separação explícita de responsabilidades entre equipas funcionais independentes. Cada equipa opera sobre uma camada específica da arquitectura e comunica exclusivamente através de contratos de interface bem definidos, responsáveis por estabelecer quais os artefactos consumidos, produzidos e disponibilizados entre componentes do sistema.
Estes contratos definem os formatos de dados, mecanismos de integração e garantias operacionais associadas a cada artefacto, incluindo schemas esperados, frequência de actualização, granularidade temporal, regras de consistência e políticas de escrita e deduplicação. Desta forma, cada equipa pode evoluir internamente os seus componentes sem afectar directamente as restantes equipas, desde que preserve o contrato acordado.
A arquitectura adopta assim um modelo de desacoplamento funcional inspirado em plataformas modernas de Data Engineering, onde as diferentes fases do pipeline operam de forma independente e substituível. Esta separação reduz o acoplamento entre componentes, simplifica a manutenção do sistema e permite escalar diferentes partes da plataforma de forma autónoma.

Equipa de Machine Learning
A equipa de Machine Learning é responsável pela componente preditiva do sistema. O seu principal objectivo consiste em utilizar os dados históricos produzidos pela camada Gold para treinar modelos de previsão de procura capazes de estimar as tendências futuras de encomendas por categoria de produto, para um horizonte temporal de sete dias.

Ao contrário das restantes equipas, esta camada não produz artefactos analíticos descritivos, mas sim artefactos inferenciais derivados de processos de treino supervisionado com Spark MLlib. O modelo — uma regressão linear com lag features de autocorrelação temporal (lag_1, lag_7, lag_14, rolling_7d_mean) e features de calendário (dia da semana, fim-de-semana, mês e trimestre) — é treinado separadamente para cada categoria de produto, utilizando os primeiros 80% das datas como conjunto de treino e os restantes 20% como conjunto de teste, em ordem cronológica estrita. As métricas de avaliação são registadas no MLflow, garantindo rastreabilidade completa de cada execução diária.

O pipeline implementa uma arquitectura de caixa analítica (analytical sandbox): as previsões são escritas primeiro num namespace de validação intermédio antes de serem promovidas para produção. A promoção é condicionada à aprovação das métricas de teste contra limiares pré-definidos — se o RMSE ou o MAE médios excederem os respectivos limiares, a tarefa de promoção falha e a tabela Gold mantém os dados do ciclo anterior, garantindo que o dashboard nunca é alimentado com resultados de um modelo degradado.

3.2 Fontes de Dados
Na ausência de uma plataforma de e-commerce real em produção, o TrendMart recorre a um simulador contínuo desenvolvido especificamente para o projeto, capaz de reproduzir o comportamento operacional de uma loja digital moderna. O simulador gera continuamente eventos de navegação, transações de compra e avaliações de clientes, permitindo alimentar o pipeline analítico com dados persistentes, correlacionados e temporalmente consistentes.
Embora os dados sejam sintéticos, a sua geração é baseada no dataset real Olist Brazilian E-Commerce (https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce), utilizado como referência para produtos, categorias, vendedores, clientes, preços e avaliações. Esta abordagem permite preservar coerência semântica entre as diferentes fontes produzidas. Os produtos visualizados durante a navegação correspondem efetivamente a produtos existentes no catálogo; as compras resultam dessas interações; e as avaliações referem-se às encomendas efetivamente concluídas. O sistema evita assim a geração arbitrária de eventos independentes, aproximando-se mais do comportamento observado em plataformas reais de e-commerce.
A definição das três fontes de dados foi guiada por três objetivos arquiteturais deliberados.
O primeiro objetivo foi a heterogeneidade estrutural. As fontes diferem intencionalmente tanto no formato dos dados como no mecanismo de ingestão utilizado: eventos semi-estruturados em streaming, registos transacionais estruturados e ficheiros de texto livre armazenados em object storage. Esta diversidade impossibilita uma abordagem puramente relacional e justifica a adopção de uma arquitectura como, Data Lake, por exemplo, com múltiplas camadas de transformação.
O segundo objectivo foi a simulação de dados contínuos e de elevado volume, particularmente ao nível do clickstream de navegação. Apesar de o simulador não conseguir correr tempo suficiente para produzir volumes verdadeiramente massivos de eventos, toda a arquitetura foi concebida para suportar cenários de ingestão contínua e crescimento progressivo dos dados ao longo do tempo. A geração contínua de eventos permite reproduzir padrões típicos de streaming observados em sistemas reais de observabilidade e análise comportamental, possibilitando validar mecanismos de ingestão incremental, processamento contínuo e particionamento temporal.
O terceiro objectivo consistiu em aproximar o sistema dos padrões operacionais encontrados em plataformas reais de e-commerce, procurando utilizar soluções tecnológicas modernas para a sua implementação. Para além da simulação dos dados de negócio, foram reproduzidos mecanismos de integração amplamente utilizados em arquitecturas de dados atuais, como streaming de eventos via Kafka, CDC em bases de dados e ingestão assíncrona em object storage. A arquitectura analítica recorre ainda a ferramentas modernas como Apache Spark para processamento distribuído, Apache Iceberg para gestão de tabelas em data lake e Trino para consulta interactiva e federada. O sistema procura assim representar não apenas os dados, mas também uma stack tecnológica realista e alinhada com práticas atuais de engenharia de dados.

3.2.1 Dados Clickstreams de Navegação

Os clickstream de navegação representam a principal fonte de observabilidade comportamental do sistema. Cada evento descreve uma interação realizada por um utilizador durante uma sessão na plataforma, permitindo reconstruir percursos completos de navegação, abandono, exploração de categorias e conversão em compra.
As sessões são geradas pelo módulo session.py através da função simulate_session(), responsável por modelar o percurso do utilizador segundo um funil probabilístico de conversão. Durante a execução de cada sessão, os eventos são produzidos sequencialmente e publicados em tempo real no tópico Kafka clickstream_events.
A publicação utiliza o session_id como chave de partição, garantindo que todos os eventos pertencentes à mesma sessão permanecem na mesma partição Kafka e preservam a sua ordem temporal relativa. Esta decisão é particularmente importante para processamento downstream, uma vez que permite reconstruir corretamente o percurso cronológico do utilizador sem necessidade de reordenação adicional.

 Tabela 13- Tipos de Eventos e Respetivas Propriedades 
      Funil de Conversão 
O comportamento do utilizador é modelado através de um funil probabilístico de conversão implementado em session.py. Cada sessão inicia obrigatoriamente com um evento session_start e evolui através de múltiplas etapas condicionais que simulam padrões realistas de navegação, hesitação, abandono e conversão.A taxa de conversão global implícita do funil é aproximadamente 10,8%, valor obtido através da multiplicação das probabilidades condicionais das etapas de conversão. Este valor encontra-se alinhado com benchmarks típicos de plataformas reais de e-commerce, contribuindo para o realismo analítico do dataset gerado.
Cada evento possui ainda um significado analítico específico dentro do funil de navegação.
session_start: início da sessão; contém o contexto geográfico do utilizador (estado e cidade)
search: navegação por intenção explícita — o termo de pesquisa é o nome da categoria do produto
category_browse: navegação por descoberta — o utilizador explora uma categoria sem intenção de pesquisa definida
product_view: visualização de um produto específico; inclui preço e número de fotografias
product_review_read: consulta de avaliações de outros compradores antes de decidir (40% dos casos)
add_to_cart: adição ao carrinho — sinal forte de intenção de compra
remove_from_cart: remoção do carrinho antes de nova adição — simula hesitação na decisão
cart_view: visualização do carrinho com o total calculado (produto + frete)
cart_abandon: abandono do carrinho sem prosseguir para checkout
checkout_start: início do processo de checkout — intenção de compra muito forte
order_placed: conclusão da compra — publicado no tópico Kafka clickstream_events e persistido no schema relacional do PostgreSQL 
session_end: encerramento da sessão com campo reason (bounce, no_cart, cart_abandon, checkout_abandon, completed)

O evento order_placed conclui logicamente o funil de conversão e é publicado no tópico Kafka clickstream_events, tal como os restantes eventos comportamentais. Em simultâneo, a função simulate_session() gera um registo de compra persistido no schema relacional do PostgreSQL através do módulo db_writer.py. Esta dualidade permite que o funil completo seja analisado diretamente a partir do clickstream, mantendo a integridade transacional na base de dados relacional.

Ruído
Para além da geração normal de eventos, o sistema introduz perturbações controladas através do módulo noise.py, com o objectivo de aproximar os dados produzidos das imperfeições observadas em ambientes reais.
Entre as anomalias simuladas encontram-se ausência de session_id (falhas de rastreamento), valores inválidos no campo device (bots, crawlers ou instrumentação incorrecta) e eventos sem event_type definido (falhas de tracking). 
A introdução deliberada de ruído permite validar a robustez dos pipelines de ingestão e transformação, evitando assumir condições idealizadas de qualidade de dados.

Armazenamento Inicial
Assim, os eventos de navegação no pipeline encontram-se inicialmente no tópico Kafka clickstream_events. Cada evento é serializado em formato JSON e armazenado como mensagem UTF-8, utilizando o session_id como chave de partição e o payload do evento como valor.
O Kafka funciona como uma camada de ingestão e transporte de eventos em tempo real, armazenando as mensagens de forma sequencial por partição e offset. 

3.2.2 Dados de Vendas

As vendas representam a componente transaccional estruturada do sistema. Uma compra concluída desencadeia duas acções em simultâneo: o evento order_placed é publicado no tópico Kafka clickstream_events como qualquer outro evento do funil, e em paralelo é gerado um registo de compra persistido no schema relacional do PostgreSQL através do módulo db_writer.py. Esta dualidade permite que o funil completo seja analisado diretamente a partir do clickstream, mantendo simultaneamente uma fonte transaccional normalizada para análise de receita e dimensões.

Armazenamento Inicial
Os pedidos são armazenados num modelo relacional composto por cinco tabelas na terceira forma normal (3NF):
Customers: um registo por cliente único; contém localização geográfica.
Sellers: um registo por vendedor único.
Products: um registo por produto único; contém categoria e número de fotografias.
Orders: tabela de factos da transacção; customer_id é chave estrangeira para customers; session_id permite correlacionar a compra com o percurso de navegação no clickstream.
Order_items: linha de item da ordem; referência orders, products e sellers por chaves estrangeiras.

Processo de Escrita
Uma compra só é gerada quando o funil de clickstream atinge a etapa order_placed — ou seja, quando o utilizador completa o checkout com sucesso. Nesse momento, a função simulate_session() devolve simultaneamente os eventos de clickstream (publicados no Kafka) e um dicionário purchase que é persistido no PostgreSQL. 
Cada compra desencadeia uma sequência de cinco operações geridas atomicamente numa única transacção pelo módulo db_writer.py: 
Upsert em customers — insere o cliente com estado e cidade; se o customer_id já existir (cliente recorrente), a operação é descartada silenciosamente (ON CONFLICT DO NOTHING).
Upsert em sellers — idêntico ao passo anterior para o vendedor.
Upsert em products — idêntico para o produto, incluindo categoria e número de fotografias.
Insert em orders — cria o registo de facto da compra, referenciando o customer_id já garantido no passo 1.
Insert em order_items — cria o item da ordem, referenciando order_id, product_id e seller_id.

O campo session_id na tabela orders preserva a ligação à sessão de navegação, permitindo a correlação downstream entre o trajecto comportamental no clickstream e a conversão efetiva na fonte transaccional.

Ruído
A fonte transaccional introduz anomalias controladas que afectam aproximadamente 19% dos registos: preços negativos ou nulos no campo price, códigos de estado mal formatados em state (diferenças de capitalização e caracteres adicionais) e categoria inválida em category.
Adicionalmente, o mecanismo de at-least-once delivery do Debezium é simulado através de noop_update_purchase(), que executa UPDATE orders SET state = state WHERE order_id = ? sem alterar qualquer valor — gerando um segundo evento CDC com op='u' para o mesmo order_id. Estes duplicados são posteriormente resolvidos na camada Silver através de deduplicação MERGE INTO sobre Apache Iceberg.





3.2.3 Avaliações de Clientes
As avaliações de clientes constituem a fonte textual semi-estruturada do sistema e representam a componente menos rígida do ecossistema de dados do TrendMart.
Ao contrário das restantes fontes, as avaliações são produzidas de forma desacoplada e assíncrona através do módulo review_writer.py, que associa reviews a aproximadamente 40% das ordens concluídas. A geração ocorre com um atraso aleatório entre 30 e 300 segundos após a compra, simulando um intervalo entre recepção do produto e submissão de feedback pelo cliente.

Estrutura do Ficheiro
O conteúdo das avaliações consistem exclusivamente em texto livre em português, sem utilização de campos delimitados ou schemas explícitos. Os únicos metadados estruturados encontram-se codificados no nome do ficheiro, obrigando o pipeline downstream a combinar processamento textual com extração de metadados contextuais.

O texto termina sempre com uma classificação explícita do produto sob uma de duas formas padronizadas:
"Dou X estrelas em 5."
"Classifico este produto com X/5."
Este padrão permite extrair automaticamente o rating através de expressões regulares na camada Silver, preservando simultaneamente o conteúdo textual integral para processamento posterior.
O atributo raw_content é mantido integralmente na camada Bronze, permitindo futuras extensões relacionadas com análise de sentimento, NLP ou classificação textual.





3.3 Área de Preparação de Dados
A área de preparação de dados constitui a componente intermédia do pipeline responsável por transformar dados operacionais heterogêneos em estruturas analíticas consistentes e semanticamente enriquecidas. O seu papel é actuar como camada de mediação entre os mecanismos de ingestão das fontes e o modelo dimensional consumido pelas equipas analíticas e de machine learning.
No contexto do TrendMart, esta área recebe continuamente três tipos de dados estruturalmente distintos — eventos semi-estruturados de navegação, alterações transacionais capturadas por CDC e ficheiros não estruturados de texto livre — e converte-os progressivamente em tabelas analíticas com schema fixo, tipagem forte e garantias de consistência transaccional.
A implementação desta área é da responsabilidade da Equipa de Engenharia de Dados, cujo contrato operacional estabelece explicitamente os limites da sua intervenção no pipeline. Como entrada, a equipa consome exclusivamente dados brutos provenientes das três fontes do simulador; como saída, entrega tabelas Silver limpas, duplicadas e semanticamente normalizadas, prontas a suportar a construção da camada Gold.
A preparação de dados encontra-se organizada segundo uma arquitectura incremental em múltiplas camadas, onde cada etapa introduz um novo nível de estrutura, qualidade e significado analítico. Esta separação permite desacoplar ingestão, preservação histórica, validação e transformação semântica, reduzindo dependências directas entre mecanismos de captura de dados e consumo analítico downstream.


3.3.1 Justificação da Abordagem
A área de preparação de dados é implementada segundo uma arquitectura Data Lakehouse baseada no padrão Medalhão (Bronze / Silver / Gold), utilizando armazenamento object-based sobre MinIO, tabelas Apache Iceberg e processamento distribuído com Apache Spark.
O modelo Lakehouse combina duas propriedades tradicionalmente separadas em arquitecturas de dados. Por um lado, herda do Data Lake a capacidade de receber dados heterogéneos no seu formato original sem necessidade de schema rígido na entrada. Por outro, incorpora características típicas de Data Warehouse, incluindo schema enforcement, garantias ACID, controlo transacional e capacidade de consulta analítica via SQL.
No TrendMart, esta separação manifesta-se explicitamente através das diferentes camadas do pipeline. A camada Bronze corresponde à componente “lake”, preservando os dados no estado mais próximo possível do original. As camadas Silver e Gold correspondem à componente “house”, onde os dados passam a obedecer a schemas explícitos, regras de qualidade, tipagem forte e estruturas optimizadas para consumo analítico.
A adopção desta arquitectura resulta directamente da heterogeneidade das fontes utilizadas pelo sistema. O pipeline recebe simultaneamente eventos JSON publicados em Kafka, envelopes CDC produzidos pelo Debezium e ficheiros de texto livre armazenados em object storage. Estas fontes apresentam níveis distintos de estrutura, granularidade e estabilidade de schema, tornando inviável a ingestão directa para um modelo relacional rígido sem uma etapa intermédia de normalização.
Um sistema puramente relacional exigiria schemas definidos à entrada e forte homogeneidade estrutural entre fontes, condições incompatíveis com a variabilidade do envelope CDC e com o caráter não estruturado das avaliações textuais. A arquitectura Lakehouse resolve este problema ao desacoplar ingestão e estruturação semântica: os dados entram primeiro no Bronze sem imposição de schema analítico e apenas posteriormente são validados, tipados e enriquecidos na camada Silver.
A separação progressiva em camadas introduz ainda vantagens arquitecturais importantes.
A primeira é a rastreabilidade histórica. Como os dados originais permanecem preservados no Bronze, qualquer alteração futura às regras de transformação pode ser aplicada retroativamente através de reprocessamento, sem necessidade de re-ingestão das fontes.
A segunda é a independência operacional. Os jobs Spark Structured Streaming mantêm checkpoints persistentes com o histórico dos ficheiros já processados, permitindo reiniciar pipelines após falhas sem risco de duplicação de dados.
A terceira é a separação de responsabilidades entre ingestão e qualidade de dados. Os consumers de ingestão limitam-se a preservar eventos no Bronze, sem conhecimento das regras semânticas aplicadas posteriormente na Silver. Esta independência reduz o acoplamento entre componentes e torna as diferentes etapas do pipeline evolutivamente substituíveis.


3.3.2 Camada Bronze

A camada Bronze constitui o ponto de entrada de todos os dados no Data Lakehouse e representa a componente de preservação histórica do sistema. O seu objectivo não é transformar, validar ou enriquecer dados, mas sim garantir que todos os eventos produzidos pelas fontes são armazenados de forma persistente, rastreável e o mais próxima possível do estado original.
Esta camada funciona como a fonte de verdade imutável do pipeline. Independentemente das regras de qualidade aplicadas posteriormente, os dados originais permanecem sempre disponíveis para auditoria, reprocessamento ou reconstrução de tabelas analíticas futuras.
Os únicos campos adicionados durante a ingestão correspondem a metadados operacionais de rastreabilidade:
ingested_at, correspondente ao instante de recepção do registo;
source, identificando o mecanismo de ingestão utilizado (kafka, debezium ou minio).
Nenhuma transformação semântica é aplicada nesta fase. Esta decisão é deliberada: impor regras de qualidade demasiado cedo aumentaria o acoplamento entre ingestão e processamento analítico, dificultando reprocessamentos futuros e reduzindo flexibilidade arquitetural.
A existência da camada Bronze desacopla ainda dois ritmos distintos do sistema: o ritmo de armazenamento dos dados e o ritmo de processamento analítico. Os mecanismos de ingestão podem continuar a escrever continuamente no Bronze independentemente da disponibilidade ou estado dos jobs Silver, tornando as duas etapas operacionalmente independentes.
	A análise de qualidade dos dados brutos, realizada sobre a camada bronze antes de qualquer transformação, revelou problemas transversais às três fontes de dados: encomendas em PostgreSQL, eventos de clickstream em Parquet e avaliações em ficheiros de texto.

Na fonte de encomendas, cerca de 17% dos registos apresentam pelo menos um problema de qualidade, o que resulta numa taxa de validade de apenas 83%. O problema mais frequente é o formato incorreto do campo de estado geográfico, que afeta aproximadamente 6,5% do total, seguido por cerca de 5,9% de registos com espaços extra no nome da categoria, o que compromete a correspondência correta com outras tabelas e pode distorcer análises agregadas. Acresce que aproximadamente 4,2% das encomendas têm preço igual ou inferior a zero, violando uma regra fundamental de negócio e tornando inválido qualquer cálculo de receita baseado nesses registos sem prévia correção.
No clickstream, a taxa de validade sobe para 90%, mas persistem três tipos de problemas relevantes. O mais significativo é a presença de cerca de 4,2% de eventos gerados por dispositivos inválidos como bots, crawlers e televisões, que introduzem tráfego não humano e podem inflar métricas como visualizações de produto e sessões iniciadas. Aproximadamente 3,2% dos eventos não têm session_id, o que impossibilita a reconstrução de percursos de navegação e a análise de funis de conversão nesses casos. Adicionalmente, cerca de 2,3% dos eventos carecem de event_type, tornando impossível classificar a ação realizada pelo utilizador.
As avaliações em texto são a fonte com mais tipos distintos de problemas. Quase 7% dos ficheiros são duplicados, e a grande maioria desses duplicados partilha o mesmo identificador de avaliação com outro ficheiro, o que pode levar a contagem dupla na análise de sentimento. Para além disso, uma pequena percentagem de registos não tem identificador de avaliação, de cliente ou de encomenda, tornando-os total ou parcialmente inúteis para cruzamento com outras fontes. Existem ainda avaliações com rating fora do intervalo válido de 1 a 5, representando cerca de 4,3% dos casos, e uma pequena fração de ficheiros com corrupção de encoding por conversão incorreta de UTF-8 para Latin-1, produzindo caracteres ilegíveis no texto das mensagens.
Em conjunto, estas anomalias justificam a existência de uma camada de limpeza Silver, onde preços inválidos, categorias mal formatadas, estados com formato errado, eventos de bots e duplicados de avaliações devem ser tratados antes de qualquer processamento analítico ou de aprendizagem automática.



Dados de clickstream de navegação

Origem
Os eventos de clickstream têm origem no módulo simulator/producer.py, que publica cada evento no tópico Kafka clickstream_events em formato JSON. O Kafka atua como buffer durável entre o simulador e o consumer de ingestão, garantindo que nenhum evento se perde mesmo que o consumer esteja temporariamente indisponível.

Extração e Transporte
Para extrair e transportar os eventos o módulo ingestion/consumer.py subscreve o tópico Kafka e consome as mensagens em modo contínuo. Cada mensagem JSON é desserializada e os seus campos são escritos diretamente no Bronze — à excepção do campo properties, que é um objecto JSON aninhado cuja estrutura varia consoante o event_type. Como o Parquet não suporta objetos de estrutura variável, o campo é serializado de volta para string Json antes de ser escrito. A desserialização fica reservada para a camada Silver, onde o Spark aplica um schema fixo via from_json. 

Armazenamento
Os eventos são escritos em ficheiros Parquet no sub-bucket bronze/clickstream/, com particionamento hierárquico em quatro níveis: year, month, day, hour. O schema fixo do ficheiro contém os campos event_id, session_id, user_id, event_type, timestamp, device, properties e ingested_at. 

Fonte 2 — Ordens de Compra

Origem
As ordens de compra começam escritas no modelo relacional do PostgreSQL Amazon Sales (customers, sellers, products, orders, order_items).
 
Extração
O Debezium monitoriza continuamente o Write-Ahead Log (WAL) do PostgreSQL e captura cada mutação das tabelas orders e order_items — INSERT, UPDATE ou snapshot inicial — publicando-as nos tópicos Kafka debezium.public.orders e debezium.public.order_items, respectivamente, encapsuladas num envelope JSON com os campos before, after, op e ts_ms. Este mecanismo de Change Data Capture permite capturar a totalidade do ciclo de vida de cada registo sem qualquer alteração ao schema ou ao código do simulador.
As tabelas de dimensão customers, sellers e products não são capturadas pelo CDC — o Debezium monitoriza exclusivamente orders e order_items. Os atributos relevantes de dimensão (estado, categoria) já viajam desnormalizados dentro desses registos, e as dimensões formais são construídas mais tarde na camada Gold a partir dos dados Silver. 
 
Transporte
O módulo ingestion/cdc_consumer.py subscreve os dois tópicos Debezium (debezium.public.orders e debezium.public.order_items) e desembrulha o envelope CDC para extrair o campo after, encaminhando cada mensagem para o buffer correspondente (orders_buf ou order_items_buf) com base no campo source.table. São processadas as operações c (INSERT), u (UPDATE) e r (snapshot inicial); os eventos d (DELETE) são descartados porque o modelo de dados do simulador não produz eliminações de ordens. O campo cdc_ts_ms é preservado no registo Parquet para que a camada Silver consiga ordenar os eventos e desduplicar pelo mais recente em caso de múltiplos UPDATEs ao mesmo order_id.  A inclusão dos UPDATEs no Bronze é intencional: o simulador executa uma operação UPDATE sem alteração de dados (noop_update) para simular o comportamento de at-least-once delivery do Debezium, gerando um segundo evento com o mesmo order_id; se esses UPDATEs fossem descartados no Bronze, o mecanismo de deduplicação MERGE INTO da camada Silver não teria oportunidade de actuar.
 
Armazenamento
Os registos são escritos em ficheiros Parquet nos sub-buckets bronze/orders/ e bronze/order_items/, com particionamento hierárquico em quatro níveis: year, month, day, hour. O schema de orders contém order_id, customer_id, session_id, purchase_timestamp, state, cdc_ts_ms e ingested_at; o de order_items contém order_item_id, order_id, product_id, seller_id, price, freight_value, category, cdc_ts_ms e ingested_at. 


Fonte 3 — Avaliações de Clientes

Origem
As avaliações estão escritas como ficheiros de texto no bucket MinIO raw-reviews/. Cada ficheiro tem o nome {review_id}_{order_id}.txt — os únicos metadados estruturados associados à avaliação estão codificados no nome do ficheiro, não no seu conteúdo.

Extracção e transporte
O módulo ingestion/file_watcher.py inspeciona o bucket raw-reviews/ a cada 30 segundos via API S3 compatível com MinIO. Para cada ficheiro .txt ainda não processado — identificado por comparação com o conjunto de object_keys registados em .watcher_state.json — o conteúdo completo é lido e incluído como campo raw_content no registo Parquet, preservando o texto livre intacto para extracção de informação na camada Silver. Após a escrita do batch, o object_key de cada ficheiro processado é adicionado ao ficheiro de estado, garantindo que o mesmo ficheiro nunca é ingerido duas vezes mesmo após reinicialização do processo.

Armazenamento
As avaliações são escritas em ficheiros Parquet no sub-bucket bronze/reviews/, com particionamento em três níveis: year, month, day — um nível a menos do que as outras duas fontes, dado o volume mais reduzido e a ausência de granularidade horária no mecanismo de polling. O schema de cada ficheiro contém os campos file_path, review_id, order_id, raw_content e ingested_at, sendo o raw_content o texto livre completo do ficheiro sem qualquer transformação. 

Métodos Comuns

Particionamento hierárquico no padrão Hive
O particionamento hierárquico constitui o principal mecanismo de optimização física utilizado na camada Bronze. Os ficheiros Parquet são organizados segundo a convenção Hive (campo=valor), reconhecida nativamente pelo Spark e pelo PyArrow.
O objectivo principal desta estratégia é permitir partition pruning durante a leitura. Quando um job Silver necessita apenas de dados pertencentes a uma determinada janela temporal, o Spark consegue eliminar antecipadamente todas as partições irrelevantes sem sequer abrir os respectivos ficheiros.
Esta otimização reduz significativamente operações de I/O, desserialização e scanning de metadata, melhorando diretamente o desempenho dos jobs downstream.
O clickstream e as ordens utilizam particionamento em quatro níveis (year/month/day/hour) devido ao maior volume e granularidade temporal contínua. As avaliações utilizam apenas três níveis (year/month/day), uma vez que o menor volume de dados não justifica subdivisão horária adicional.


Micro-batching
A ingestão na camada Bronze utiliza uma estratégia de micro-batching, onde múltiplos registos são acumulados temporariamente em memória antes de serem escritos em ficheiros Parquet únicos.
Esta abordagem procura equilibrar dois requisitos contraditórios.
Por um lado, ficheiros demasiado pequenos degradam o desempenho do Spark devido ao elevado overhead de metadata e abertura de ficheiros. Por outro, batches demasiado grandes aumentam a latência entre a geração do evento e a sua disponibilidade analítica.
Os consumers Kafka e CDC resolvem este equilíbrio através de dois critérios alternativos de flush:
escrita imediata após 500 registos;
Escrita após 30 segundos sem flush.
O primeiro critério optimiza eficiência de leitura sob carga elevada; o segundo garante baixa latência durante períodos de tráfego reduzido.
O file watcher das avaliações segue um modelo ligeiramente distinto: cada ciclo de polling produz diretamente um batch correspondente ao conjunto de novos ficheiros detectados no bucket MinIO.


3.3.3 Camada Silver

A camada Silver representa o ponto de transição entre armazenamento bruto e estrutura analítica governada. É nesta etapa que os dados deixam de ser apenas artefactos persistentes e passam a obedecer explicitamente a regras de qualidade, schemas analíticos e garantias de consistência transaccional.
Enquanto a camada Bronze privilegia preservação e rastreabilidade, a camada Silver privilegia consistência semântica e fiabilidade analítica.
Cada job Silver lê os ficheiros Parquet correspondentes da camada Bronze, aplica validações específicas ao domínio de dados respectivo e escreve o resultado em tabelas Apache Iceberg registadas no catálogo lake.
É nesta camada que se concentram:
validação de schema;
tipagem forte;
normalização semântica;
enriquecimento contextual;
deduplicação;
filtragem de ruído;
garantias ACID sobre escrita incremental.
As tabelas produzidas nesta etapa constituem a base operacional consumida posteriormente pela Equipa de Engenheiros Analíticos para construção do modelo dimensional Gold.





Job 1 — silver_clickstream.py

Entrada
Ficheiros Parquet de bronze/clickstream/, lidos com schema explícito para evitar inferência automática.

Transformações e regras de qualidade
O campo properties é desserializado para struct com schema definido, extraindo product_id, category, price, state, city, query e reason. Destes, apenas category, product_id e price são promovidos a colunas individuais na saída; state e city são fundidos num único campo location via coalesce (estado prioritário sobre cidade); query e reason são descartados.
Os eventos com event_id, session_id ou event_type nulos ou vazios são descartados, assim como os que não produzem um event_ts válido após conversão do timestamp.
Os valores do campo device são normalizados para o vocabulário controlado {mobile, desktop, tablet}: valores não reconhecidos são normalizados para "unknown". 

Armazenamento
Tabela Iceberg lake.silver.clickstream com schema fixo e tipagem forte, em modo append. O schema da tabela contém os campos event_id, session_id, user_id, event_type, event_ts (timestamp tipado), device, category, product_id, price, location e ingested_at. 

Job 2 — silver_orders.py

Entrada
Dois conjuntos de ficheiros Parquet: bronze/order_items/ (stream principal — contém price, freight_value, product_id, seller_id e category) e bronze/orders/ (leitura em batch por foreachBatch — contém customer_id, session_id, purchase_timestamp e state). Os dois DataFrames são joined em order_id antes de qualquer transformação.

Transformações e regras de qualidade
Os campos price e freight_value são convertidos de string para DOUBLE e os registos com price ≤ 0 ou freight_value < 0 são descartados.
O purchase_timestamp — enviado pelo Debezium em microssegundos desde epoch — é convertido para Timestamp via divisão por 1 000 000; registos com timestamp inválido são descartados.
O campo category é normalizado com TRIM para remover espaços excedentes.
O campo state é validado contra os 27 códigos de estado brasileiros: variantes com casing errado ("sp"), espaços (" SP ") ou typos ("SP0") resultam em state = NULL e region = "Desconhecido", sem descartar a ordem.
O total_value é calculado como round(price + freight_value, 2).
Cada ordem é enriquecida com o campo region, derivado do mapeamento dos 27 estados para as cinco macrorregiões brasileiras. 

Armazenamento
Tabela Iceberg lake.silver.orders. O schema contém os campos order_id, session_id, customer_id, product_id, seller_id, category, price, freight_value, total_value, purchase_ts, state, region e ingested_at. 





Job 3 — silver_reviews.py

Entrada
Ficheiros Parquet de bronze/reviews/, que contêm os campos review_id, order_id, raw_content e ingested_at. Os campos review_id e order_id foram extraídos do nome do ficheiro durante a ingestão Bronze — chegam já como colunas tipadas. 

Transformações e regras de qualidade
Registos com review_id ou order_id nulos ou vazios são descartados.
O rating é extraído do texto livre por duas expressões regulares — "Dou X estrelas em 5" e "Classifico este produto com X/5"; registos sem match válido no intervalo [1, 5] são descartados.
O campo message é criado por TRIM(raw_content); registos com mensagem vazia após o trim são descartados.
O text_length é calculado como contagem de palavras por split em espaços. 

Armazenamento
Tabela Iceberg lake.silver.reviews. O schema contém os campos review_id, order_id, rating (INT), message, text_length e ingested_at. 

Mecanismos Comuns

Checkpoint do Spark Structured Streaming
Os três jobs utilizam trigger(availableNow=True), que instrui o Spark a processar todos os ficheiros Bronze ainda não consumidos e a terminar — em vez de correr indefinidamente como um stream contínuo. O estado de progresso é mantido num checkpoint persistido em MinIO (s3a://silver/_checkpoints/{clickstream,orders,reviews}): o Spark regista neste checkpoint os ficheiros Bronze já processados em runs anteriores e, no início de cada novo run, lê apenas os ficheiros ainda não registados. Este mecanismo garante independência: se um job falhar a meio e for relançado pela DAG, não reprocesse dados já escritos na Silver.

Deduplicação com MERGE INTO Iceberg
Os jobs de ordens e reviews enfrentam um problema adicional de deduplicação: o Bronze pode conter múltiplos eventos para o mesmo order_id (devido ao at-least-once delivery do Debezium) ou para o mesmo review_id (devido às submissões duplicadas injectadas pelo simulador). A deduplicação é feita em dois níveis. Dentro de cada micro-batch processado pelo Spark, uma window function (ROW_NUMBER OVER PARTITION BY order_id/review_id ORDER BY cdc_ts_ms DESC) retém apenas o evento mais recente por chave. Entre runs, o MERGE INTO Iceberg com WHEN NOT MATCHED THEN INSERT * garante que um order_id ou review_id já existente na Silver nunca é duplicado por runs posteriores — independentemente do número de vezes que o mesmo evento apareça no Bronze. O job de clickstream não necessita deste mecanismo porque os eventos de clickstream são naturalmente únicos por event_id e são escritos em modo append directo.

O código dos três jobs é da responsabilidade da Equipa de Engenharia de Dados; a execução é agendada e despoletada pela DAG trendmart_gold_pipeline da Equipa de Engenharia de Infraestrutura (infrastructure/dags/dag_trendmart.py), com cadência horária definida como requisito de SLA pela Equipa de Engenharia Analítica.


3.4 Área de Dados do Sistema de Análise
A camada Gold constitui a componente analítica do pipeline, onde os dados previamente limpos na camada Silver são transformados num modelo estruturado e orientado à análise de negócio. O seu principal objetivo é garantir que os dados chegam aos consumidores já agregados, enriquecidos e organizados por dimensão analítica, eliminando qualquer necessidade de lógica de transformação no cliente.
A implementação desta camada é da responsabilidade da equipa de Engenharia Analítica. O seu contrato define a entrada como as tabelas Silver e a saída como um modelo analítico em esquema estrela, exposto sob a forma de tabelas Iceberg e vistas SQL em Trino.



3.4.1 Modelo Dimensional

O sistema de dados analítico do TrendMart implementa um modelo dimensional em esquema estrela na camada Gold do Data Lakehouse, armazenado em tabelas Apache Iceberg sobre MinIO e acessível via Trino. A escolha de um modelo dimensional — em detrimento de uma modelação relacional normalizada (3NF) — decorre dos padrões de acesso analítico dominantes: queries que atravessam múltiplas dimensões (por exemplo, "receita por categoria, por mês, por região"), contagens e agregações sobre grandes volumes de factos, e filtragens sobre atributos descritivos das dimensões. O esquema estrela minimiza o número de joins necessários por query analítica e torna as estruturas de dados imediatamente compreensíveis para analistas sem conhecimento técnico do pipeline. 

Figura 6 - Modelo Dimensional em Esquema Estrela (Camada Gold) 

O processo de conceção seguiu os quatro passos do método de Kimball, com as escolhas de design ancoradas nos requisitos funcionais do capítulo 2.

Passo 1 — Identificação dos Processos de Negócio
Foram identificados três processos de negócio distintos mas complementares, cada um correspondendo a uma tabela de factos:
Vendas e Encomendas: quanto vendemos, onde, e em que categorias? O ciclo de vida de uma encomenda desde a compra até à sua conclusão, com métricas financeiras — suporta os perfis Gestor Executivo (RF1.1–RF1.2), Analista de Vendas (RF2.1–RF2.3) e Analista de Tendências (RF5.1–RF5.3).
Comportamento de Navegação: como é que os utilizadores navegam e onde abandonam? Os eventos que os utilizadores geram na plataforma, que permitem analisar o funil de conversão digital — suporta o perfil Analista de Marketing (RF3.1–RF3.5).
Avaliações de Clientes: o que dizem os clientes sobre os produtos? O feedback qualitativo pós-compra, com classificação de sentimento — suporta o perfil Gestor de Customer Experience (RF4.1–RF4.4).


Passo 2 — Definição da Granularidade
Para a tabela de factos fact_sales, a granularidade é ao nível da ordem individual: cada linha representa uma transação de compra. Esta granularidade permite agregar métricas em qualquer nível superior (categoria, dia, região) sem perda de informação.
Para a tabela de factos fact_clickstream, a granularidade é ao nível do evento individual: cada linha representa um clique, visualização, ou ação no funil. Esta granularidade permite reconstruir sessões completas e calcular taxas de conversão em qualquer ponto do funil.
Para a tabela de factos fact_reviews, a granularidade é ao nível da avaliação individual: cada linha representa um ficheiro de review processado, com o rating, sentimento, categoria e região associados.

Passo 3 — Identificação das Dimensões
A seleção das dimensões partiu da análise dos eixos de análise exigidos pelos requisitos funcionais do capítulo 2. Os perfis de natureza ERP e o Gestor de Customer Experience necessitam de análise por categoria; o Analista de Vendas e o Gestor de Customer Experience necessitam de análise geográfica; o Analista de Marketing necessita de análise temporal enriquecida com atributos de calendário. Isto determina exactamente três dimensões, necessárias e suficientes:
dim_date: responde ao quando com atributos derivados. Criada como calendário completo de 2020 a 2030 (4 018 datas), com atributos year, month, quarter, week, day_of_week, is_weekend. É referenciada exclusivamente por fact_clickstream via date_id, justificada pelo RF3.5 — comparação do comportamento de navegação entre dias úteis e fins de semana. As tabelas fact_sales e fact_reviews utilizam as colunas de data diretamente (purchase_date e review_date), uma vez que nenhum requisito associado a essas fontes exige atributos de calendário derivados.
dim_category: responde ao o quê ao nível de categoria. Contém os nomes de categoria derivados das ordens e do clickstream, com category_id gerada por hash determinístico do nome. Partilhada pelas três tabelas de factos.
dim_geography: responde ao onde. Mapeia os 27 estados brasileiros para as cinco macrorregiões (Norte, Nordeste, Centro-Oeste, Sudeste, Sul), com geo_id gerado por hash do código de estado. Partilhada por fact_sales e fact_reviews.
Não foram criadas dimensões ao nível de produto ou de cliente porque nenhum requisito funcional do capítulo 2 exige análise a esse nível de detalhe — toda a análise de produto é feita ao nível de categoria (RF2.1, RF4.3, RF5.3) e toda a análise de cliente é expressa em contagens de clientes únicos (RF1.1, RF2.2), não em atributos de clientes individuais.


Passo 4 — Identificação dos Factos
A tabela de factos fact_sales contém métricas aditivas (price, freight_value, total_value), as chaves de negócio order_id e customer_id, e chaves estrangeiras para dim_category (category_id) e dim_geography (geo_id). O customer_id é preservado como atributo degenerado para permitir a contagem de clientes únicos (RF1.1, RF2.2). A data da compra é armazenada diretamente como purchase_date — coluna que serve simultaneamente como eixo temporal de análise e como coluna de particionamento da tabela Iceberg.
A tabela de factos fact_clickstream contém event_type e device como factos degenerados (vocabulários controlados com número fixo de valores), session_id e user_id como chaves de negócio, e chaves estrangeiras para dim_date (date_id) e dim_category (category_id). A FK para dim_date é a única no modelo — justificada pelo RF3.5, que exige comparação do comportamento de navegação entre dias úteis e fins de semana via o atributo is_weekend. Não tem FK para dim_geography porque nenhum requisito do perfil Analista de Marketing (RF3.1–RF3.5) exige análise geográfica do comportamento de navegação.
A tabela de factos fact_reviews contém rating (semi-aditivo — o indicador relevante é a média, não a soma), sentiment como facto degenerado (positive, neutral, negative), text_length como medida descritiva, e chaves estrangeiras para dim_category (category_id) e dim_geography (geo_id). O geo_id é resolvido em gold_reviews.py por join com silver.orders para obter o estado do cliente da ordem associada, seguido de lookup em dim_geography — garantindo que a geografia fica inscrita no próprio facto e não depende de um join posterior através de fact_sales. A data da avaliação é armazenada diretamente como review_date.
A tabela ml_demand_forecast é uma tabela Gold adicional que não faz parte do modelo dimensional principal mas partilha a convenção de nomenclatura e é acessível via Trino. É gerada pela DAG independente trendmart_ml_pipeline e consumida diretamente pelo dashboard sem passar por vistas.


3.4.2 Implementação Física do Schema

Os jobs Spark da equipa analytical_engineering são invocados pela DAG trendmart_gold_pipeline com cadência horária. Existem dois padrões de escrita distintos consoante o tipo de tabela.

Tabelas de Dimensões

O job de dimensões (gold_dimensions.py) recria integralmente as três tabelas — dim_date, dim_category e dim_geography — a cada execução via writeTo().createOrReplace().  Esta escolha é deliberada: as dimensões têm um conjunto de valores pequeno e estável, e recalculá-las completamente garante que qualquer nova categoria ou novo estado presente no Silver fica refletido imediatamente, sem lógica de merge ou detecção de diferenças.

Tabelas de factos

Os três jobs de factos (gold_sales.py, gold_clickstream.py, gold_reviews.py) usam um modelo de carga incremental baseado na janela de execução do Airflow. O DAG passa a cada job os parâmetros data_interval_start e data_interval_end — os timestamps exatos da hora que está a ser processada —, e o job filtra a tabela Silver pelo campo ingested_at dentro dessa janela:
ingested_at >= data_interval_start  AND  ingested_at < data_interval_end
O campo ingested_at é atribuído por-registo no momento em que cada consumer Bronze recebe a mensagem (do Kafka, do CDC ou do MinIO), pelo que reflecte o instante real de ingestão de cada evento individual — não o timestamp do ficheiro Parquet que o contém.

Na primeira execução, quando a tabela Gold ainda não existe, o job usa writeTo().createOrReplace() para a criar com o schema e particionamento corretos. Nas execuções seguintes usa writeTo().append(), adicionando apenas os registos da hora processada. Se não existirem registos na janela — por exemplo num período de tráfego nulo —, o job termina sem escrever, evitando commits vazios.
Este modelo é distinto do mecanismo incremental da camada Silver: os jobs Silver usam Spark Structured Streaming com trigger(availableNow=True) e um checkpoint persistido em MinIO, que rastreia quais ficheiros Bronze já foram lidos pelo path — sem dependência do Airflow. O Gold delega essa responsabilidade ao orquestrador, o que torna cada run Gold determinístico e reproduzível: re-executar a janela das 13:00–14:00 produz sempre o mesmo resultado, independentemente de quando a re-execução ocorre.

3.5 O Processo de Integração de Dados
 
O processo de integração de dados do TrendMart segue o paradigma ELT (Extract, Load, Transform): os dados são primeiro extraídos das fontes e carregados no Data Lake Bronze no seu formato original, e só depois transformados nas camadas Silver e Gold. Esta abordagem preserva os dados originais indefinidamente como fonte de verdade imutável, desacopla o ritmo de ingestão do ritmo de transformação e permite reprocessar toda a cadeia a partir do Bronze sem re-ingestão das fontes.
 
3.5.1 Modelo do Processo de Integração (BPMN)
 
O processo de integração de dados é modelado segundo a notação BPMN, com três pools sequenciais correspondentes às três camadas do Data Lake: o Pool Bronze, responsável pela ingestão contínua das fontes; o Pool Silver, responsável pela transformação e validação Bronze → Silver via Spark; e o Pool Gold, responsável pela integração no modelo dimensional e criação das vistas Trino. O diagrama completo encontra-se no Anexo A. 

Pool Bronze — Ingestão Contínua
O Pool Bronze agrupa três lanes paralelas, uma por fonte de dados, cada uma com um consumer dedicado que corre continuamente e de forma independente da orquestração Airflow. O consumer Kafka subscreve o tópico clickstream_events e acumula eventos em memória até ao limiar de 500 registos ou 30 segundos, escrevendo então um ficheiro Parquet na partição Bronze correspondente. O consumer CDC subscreve os dois tópicos Debezium (debezium.public.orders e debezium.public.order_items) e aplica o mesmo mecanismo de micro-batching às ordens de compra capturadas do PostgreSQL. O file watcher consulta o bucket raw-reviews/ a cada 30 segundos, identifica ficheiros novos por comparação com o estado persistido em .watcher_state.json, e escreve os conteúdos em Parquet Bronze. Os três processos correm em paralelo e não têm dependências entre si.

 
Figura 7-

Pool Silver — Transformação Batch (Airflow + Spark)
O Pool Silver é orquestrado pelo Airflow com execução horária. As três tarefas Silver executam em paralelo, cada uma processando os ficheiros Bronze disponíveis desde o último ciclo através de Spark Structured Streaming com trigger(availableNow=True).
Cada job aplica as regras de validação, limpeza e enriquecimento descritas na secção 3.3.3 e escreve os resultados nas tabelas Iceberg Silver correspondentes. A idempotência é assegurada por dois mecanismos distintos: o silver_clickstream garante exactly-once através do sink nativo do Iceberg combinado com checkpoint Spark; os jobs silver_orders e silver_reviews usam foreachBatch com operação MERGE INTO para deduplicação cross-batch. 
 
Pool Gold — Carga Analítica (Airflow + Spark)
O Pool Gold inicia após a conclusão das três tasks Silver. A task gold_dimensions executa primeiro, recriando as três dimensões com createOrReplace. Após a sua conclusão, as tasks gold_sales, gold_clickstream e gold_reviews executam em paralelo, fazendo append incremental às tabelas de factos para a janela horária processada. Por fim, a task init_views actualiza as seis vistas Trino, tornando os dados imediatamente disponíveis para o dashboard.

3.5.3 Ferramentas
 
As ferramentas que compõem o pipeline foram seleccionadas em conjunto pela equipa de infraestrutura e pelas equipas utilizadoras de cada componente, com base em critérios de compatibilidade técnica entre si, maturidade em ambientes de produção e adequação ao volume e natureza de dados do sistema. Todas são open-source e correm integralmente em Docker Compose.
 
Apache Kafka
O Apache Kafka é a plataforma de streaming utilizada como buffer durável para os eventos de clickstream. A sua adopção justifica-se por três requisitos que alternativas mais simples não satisfazem em conjunto.
O primeiro é a tolerância a falhas do consumer. Sem Kafka, a opção mais directa seria o simulador escrever eventos directamente para MinIO — mas se o consumer estiver em baixo quando os eventos chegam, esses eventos perdem-se sem possibilidade de recuperação. O Kafka persiste as mensagens em disco independentemente do estado do consumer: quando o consumer reinicia, retoma a partir do último offset confirmado sem perda de dados.
O segundo é a ordenação garantida por sessão. A análise de funil depende de conhecer a sequência de eventos dentro de cada sessão — se o utilizador viu o produto antes de comprar, quanto tempo decorreu entre a visualização e o abandono do carrinho. Sem ordenação garantida, estas análises produzem resultados incorrectos. O Kafka garante ordenação dentro de cada partição, e usando session_id como chave de partição, todos os eventos de uma sessão chegam ao consumer na ordem em que foram gerados.
O terceiro é o desacoplamento entre produção e consumo. O simulador publica eventos ao seu próprio ritmo, independentemente de o consumer estar disponível ou de quão rapidamente processa. Sem este desacoplamento, uma lentidão ou falha no consumer bloquearia ou atrasaria a geração de dados.
As alternativas consideradas não satisfazem estes três requisitos em conjunto. O RabbitMQ é uma queue de mensagens tradicional adequada para task queues, mas apaga as mensagens após consumo sem replay em caso de falha e não tem ordenação por partition key. O Redis Streams é mais leve mas com persistência menos robusta e sem o conceito de partition key para garantir ordenação por sessão. A escrita directa para MinIO elimina o buffer e a tolerância a falhas do consumer, tornando a ingestão dependente da disponibilidade contínua de ambos os processos em simultâneo.
 
Debezium
O Debezium é o conector de Change Data Capture utilizado para capturar as alterações nas tabelas orders e order_items do PostgreSQL. Monitoriza o Write-Ahead Log (WAL) do PostgreSQL e publica cada INSERT, UPDATE ou snapshot inicial como um evento estruturado nos tópicos Kafka debezium.public.orders e debezium.public.order_items. A sua adopção justifica-se por não requerer qualquer alteração ao schema ou ao código da base de dados fonte — a captura é transparente para o simulador — e por preservar o histórico completo de mutações, incluindo os noop updates que permitem ao MERGE INTO da camada Silver actuar sobre duplicados.
 
MinIO
O MinIO é a plataforma de object storage S3-compatível que serve de substrato ao Data Lake, persistindo os ficheiros Parquet da camada Bronze, os checkpoints Spark e os ficheiros de avaliações em texto. A alternativa mais óbvia seria o AWS S3, mas implicaria dependência de serviços cloud, custos de utilização e impossibilidade de execução local  incompatível com um sistema que corre integralmente em Docker Compose. A alternativa local mais simples seria o sistema de ficheiros do próprio sistema operativo, mas o Spark acede ao storage via conector S3A, o PyArrow usa o protocolo S3, e o file watcher usa boto3  todos assumem uma API S3. Usar o filesystem local obrigaria a adaptações em cada um destes componentes ou à substituição dos conectores. O HDFS seria a alternativa clássica em arquitecturas Hadoop, mas é significativamente mais pesado para correr em Docker e introduz complexidade operacional desproporcional ao volume de dados do sistema. O MinIO resolve o problema com uma instalação Docker simples que expõe exactamente a API S3 que todos os componentes já esperam, sem qualquer adaptação, e com suporte ao particionamento Hive que o Spark usa para eliminação de partições na leitura. 
 
Apache Spark
O Apache Spark é o motor de processamento distribuído utilizado para todas as transformações Silver e Gold. A sua adopção justifica-se pela integração nativa com o Apache Iceberg  via Iceberg Spark Runtime  que permite operações ACID como MERGE INTO e append transaccional directamente sobre o Data Lake; pela capacidade de Structured Streaming com trigger(availableNow=True), que processa todos os ficheiros Bronze pendentes num único run e termina, adequado ao modelo batch horário do sistema; e pela escalabilidade horizontal, que permite aumentar o volume de dados processado por run sem alteração do código dos jobs.
A alternativa mais directa seria Python com Pandas. No entanto, Pandas não tem equivalente ao MERGE INTO: para deduplicar contra a tabela Silver existente, seria necessário ler a tabela inteira para memória, concatenar com os dados novos, deduplicar e reescrever tudo  uma operação que não escala à medida que a Silver cresce e que, a meio de uma reescrita de ficheiros Parquet no MinIO, não oferece qualquer garantia de atomicidade. Acresce que o rastreamento de quais ficheiros Bronze já foram processados teria de ser implementado de raiz, replicando manualmente o que o sistema de checkpoints do Spark oferece nativamente. O Spark resolve os três problemas  upsert eficiente, atomicidade das escritas e rastreamento de fonte  com mecanismos integrados e testados em produção. 
 
Apache Iceberg
O Apache Iceberg é o formato de tabela open-source utilizado nas camadas Silver e Gold, adotado não por conveniência, mas por responder a três requisitos fundamentais da arquitetura.
O primeiro requisito é a gestão robusta de falhas em produção. Aqui, o sistema combina checkpoints do Spark com o Iceberg de forma complementar. Os checkpoints garantem que os ficheiros do Bronze não são reprocessados em caso de retry do Airflow, enquanto o Iceberg assegura que escritas incompletas (como MERGE INTO ou append interrompidos) não ficam visíveis na tabela. Esta combinação garante consistência end-to-end: a fonte é consumida exatamente uma vez e o destino só é atualizado se a operação for concluída na totalidade. Problemas como processos órfãos causados por SIGKILL obrigaram ainda à implementação de mecanismos de limpeza de checkpoints antes de novas execuções.
O segundo requisito é o suporte a operações de upsert sobre dados CDC. O Debezium gera eventos duplicados por natureza (snapshot + inserts/updates), o que invalida abordagens simples de append. O Iceberg permite resolver este problema com MERGE INTO atómico, permitindo atualizar ou inserir registos com base em chaves como order_id ou review_id, sem necessidade de reprocessar tabelas completas ou introduzir lógica manual de deduplicação.
O terceiro requisito é a eficiência de leitura na camada analítica. As tabelas Gold são particionadas por data e o Iceberg mantém metadados que apontam diretamente para os ficheiros físicos relevantes. Isto permite ao Trino fazer pruning de partições e ler apenas dados relevantes em queries temporais, reduzindo drasticamente o custo de leitura face a uma varredura completa de ficheiros Parquet.
Foram consideradas alternativas como Delta Lake e Apache Hudi, que também suportam ACID e MERGE INTO. No entanto, o Delta apresenta maior acoplamento ao ecossistema Databricks e menor maturidade de integração com Trino, enquanto o Hudi adiciona complexidade operacional (Copy-on-Write vs Merge-on-Read) sem benefício claro para o volume do sistema. O Iceberg destacou-se por oferecer integração sólida e simultânea com Spark e Trino, alinhando-se melhor com os requisitos multi-engine da arquitetura.

Apache Trino
Numa versão anterior da arquitectura, a camada analítica assentava num esquema clássico de data warehouse: os dados das tabelas de factos e dimensões eram materializados no PostgreSQL através de um passo de sincronização dedicado, e o dashboard consultava directamente essa base de dados relacional. Esta abordagem introduzia um problema estrutural: qualquer desfasamento entre o pipeline e a sincronização para o PostgreSQL criava inconsistências na camada de apresentação, e o passo de materialização constituía uma fonte de falha autónoma. Quando a equipa decidiu consolidar toda a camada analítica no lakehouse  eliminando o PostgreSQL como destino intermédio e passando a escrever directamente em tabelas Iceberg sobre MinIO  o problema foi resolvido, mas um novo emergiu: como servir queries interactivas ao dashboard a partir de ficheiros Parquet em object storage?
O Apache Trino resolve este problema de forma directa. Como motor de consulta SQL distribuído com suporte nativo ao formato Iceberg, o Trino acede às tabelas Gold através do mesmo Hive Metastore que o Spark utiliza para escrita. Esta arquitectura de catálogo partilhado elimina qualquer passo de sincronização: assim que o Spark conclui a escrita de uma tabela Gold, o Trino vê imediatamente os novos dados sem qualquer intervenção adicional. Os ficheiros Parquet nunca saem do MinIO  o Trino lê-os directamente, respeitando o esquema Iceberg e o particionamento físico das tabelas, o que reduz o I/O em queries com filtros temporais.
A alternativa natural seria reutilizar o Spark como motor de consulta do dashboard. No entanto, o Spark foi concebido como motor de transformação batch, não como query engine interactivo: o tempo de inicialização de um job Spark situa-se entre 15 e 30 segundos, o que tornaria cada refresh do dashboard perceptivelmente lento e degradaria a experiência de utilização. O Trino, por contraposição, executa queries de agregação sobre vistas pré-calculadas em segundos, sem overhead de inicialização — adequado ao modelo de actualização de cinco em cinco minutos da aplicação Dash.
Para além da performance de leitura, o Trino cumpre um segundo papel arquitectural: servir de camada de abstracção entre o pipeline e o dashboard através de vistas SQL. As seis vistas analíticas  vw_executive, vw_sales_performance, vw_funnel, vw_reviews, vw_trends e vw_category_trends  encapsulam toda a lógica de joins, agregações e cálculos derivados sobre o modelo dimensional. O dashboard nunca acede directamente às tabelas de factos ou dimensões; consome exclusivamente as vistas. Esta separação tem uma consequência prática relevante: alterações no modelo dimensional — mudanças de nomes de colunas, adição de métricas, reformulação de joins — são absorvidas pelas vistas sem qualquer impacto no código do dashboard. As mesmas vistas podem ainda ser consumidas por outras ferramentas de Business Intelligence externas que suportem JDBC ou a API Trino, assegurando a interoperabilidade da camada analítica.

 Hive Metastore
O Hive Metastore é o serviço de catálogo que regista o schema, a localização física e os metadados de particionamento de todas as tabelas Iceberg do sistema. Para cada tabela Silver e Gold, o HMS mantém o mapeamento entre o nome lógico da tabela, o esquema de colunas e tipos, e o caminho no MinIO onde os ficheiros Parquet e os ficheiros de metadados Iceberg residem. É este registo que permite ao Spark e ao Trino referirem-se a lake.silver.orders ou lake.gold.fact_sales como entidades com nome, em vez de caminhos físicos hardcoded no object storage.
A adopção do Hive Metastore não foi uma imposição técnica  o Iceberg suporta catálogos alternativos, nomeadamente o REST Catalog nativo, o JDBC Catalog e o Nessie. A escolha recaiu sobre o HMS por uma razão operacional concreta: é o único catálogo com integração de primeira classe, estável e amplamente documentada, em simultâneo no Spark e no Trino. O REST Catalog do Iceberg, apesar de ser a opção mais moderna e arquitecturalmente mais limpa, apresentava suporte menos maduro nos dois motores à data de implementação, com configurações menos estáveis e documentação mais escassa. Utilizar o HMS como catálogo partilhado era a opção que minimizava o risco operacional mantendo a compatibilidade total entre os dois motores.
A consequência prática desta escolha é o catálogo único partilhado: quando o Spark escreve uma tabela Gold e regista o commit no HMS, o Trino vê imediatamente essa tabela sem qualquer passo de sincronização adicional. Os dois motores operam sobre o mesmo estado de catálogo, o que garante consistência entre a camada de transformação e a camada de consulta sem intervenção manual.

Apache Airflow
O Apache Airflow é o orquestrador de pipelines que gere a execução das transformações Silver e Gold. A alternativa mais simples seria um cron job que corre um script Python de hora a hora — mas um cron job não tem noção de dependências entre tasks. Para replicar o grafo de dependências do pipeline, o script teria de encadear manualmente chamadas com verificação de estado, gerir paralelismo com threads ou subprocessos, e implementar lógica de retry dentro do próprio código — replicando, de forma mais frágil, o que o Airflow oferece nativamente.
A adopção do Airflow justifica-se por quatro propriedades. Primeiro, a definição declarativa de dependências em DAG: o grafo determina que as tasks Silver correm em paralelo, que as tasks Gold de factos só arrancam após gold_dimensions completar, e que init_views aguarda a conclusão de todas as tasks Gold — sem qualquer lógica de sincronização manual. Segundo, retentativas automáticas por task com atraso configurável: o pipeline teve falhas reais de jobs Spark por timeout e por JVMs órfãos; sem retentativas automáticas, cada falha exigiria intervenção manual para re-executar apenas a task afectada sem repetir as que já concluíram com sucesso. Terceiro, auditabilidade: o Airflow mantém o registo de cada run — quais tasks correram, quando, quanto tempo demoraram e o log de output — o que permite diagnosticar gaps de dados sem aceder directamente às tabelas. Quarto, paralelismo declarativo: tasks independentes correm em simultâneo sem threading manual no código.
O sistema utiliza duas DAGs independentes com cadências distintas. A DAG trendmart_pipeline corre de hora a hora. 
Em caso de falha, o Airflow retenta as tasks Silver após 15 minutos e as tasks Gold após 5 minutos. A DAG trendmart_ml_pipeline corre diariamente às 03:00 UTC, após o pipeline Gold ter os dados do dia completos, e executa a task demand_forecast, que treina o modelo de previsão de procura e escreve os resultados em lake.gold.ml_demand_forecast. Em caso de falha, retenta após 10 minutos.




4. Exploração e Análise de Dados

4.1 Organização Geral do Sistema de Dashboarding
A camada de visualização do TrendMart é implementada como uma aplicação web em Plotly Dash com Bootstrap, estruturada em seis dashboards temáticos. Cada dashboard foi desenhado para responder às necessidades analíticas dos perfis de utilizador definidos no capítulo 2.
A aplicação está disponível em http://localhost:8050 e atualiza automaticamente os dados a cada cinco minutos através do componente dcc.Interval, garantindo que a informação apresentada reflecte o estado mais recente da camada Gold sem intervenção manual.

4.1.1 Arquitetura da Camada de Visualização
A aplicação está organizada em dois módulos Python com responsabilidades claramente separadas.
O módulo data.py implementa a camada de acesso a dados. Este módulo contém exclusivamente funções responsáveis por construir queries SQL parametrizadas, executá-las sobre as vistas do Trino através do cliente trino.dbapi, e devolver os resultados em DataFrames Pandas.
Cada função corresponde a um dashboard específico — get_executive, get_sales_performance, get_funnel, get_reviews, get_trends, get_category_trends e get_demand_forecast — recebendo como parâmetro a janela temporal de análise em dias. Esta separação permite isolar a lógica de acesso a dados, facilitando testes, manutenção e possível substituição da fonte de dados sem impacto na camada de apresentação.
O módulo app.py implementa a camada de apresentação. É responsável pela definição do layout da aplicação, organizado em abas (dcc.Tabs) correspondentes aos seis dashboards. Este módulo regista o callback principal da aplicação, que reage tanto à alteração de dashboard como ao evento de atualização periódica, e delega a renderização de cada vista para funções específicas (_tab_executive, _tab_sales, _tab_funnel, _tab_reviews, _tab_trends, _tab_ml).
A aplicação utiliza uma arquitetura de callback único, na qual um único @app.callback re-renderiza integralmente o dashboard ativo a cada atualização. Esta abordagem simplifica a gestão de estado, reduz a complexidade de sincronização entre componentes e garante consistência dos dados apresentados em cada refresh.
A utilização de vistas Trino como interface entre o pipeline analítico e a camada de visualização apresenta três vantagens principais. Em primeiro lugar, os dashboards recebem dados já agregados e prontos a consumir, reduzindo significativamente a complexidade de processamento em Python. Em segundo lugar, alterações no modelo dimensional são encapsuladas nas vistas, sem impacto direto na camada de visualização. Em terceiro lugar, as mesmas vistas podem ser reutilizadas por outras ferramentas de Business Intelligence, assegurando interoperabilidade da camada analítica.

5. O Sistema de Análise de Tendências de Aquisição de Produtos

5.1 Definição do problema e compreensão dos elementos de análise envolvidos
O problema central abordado neste sistema é a previsão da procura de produtos por categoria ao longo de um horizonte temporal de sete dias. Em contextos de comércio eletrónico, a antecipação da procura é determinante para a gestão eficiente de inventário, para o planeamento logístico e para a ativação proativa de alertas a vendedores. A ausência de previsões fiáveis conduz frequentemente a rupturas de stock, excesso de aprovisionamento ou falhas na capacidade de resposta ao cliente. O sistema Projeto responde a este problema através de um pipeline de machine learning integrado no ecossistema de dados da plataforma. A análise de tendências de aquisição é tratada como um problema de previsão de séries temporais por categoria de produto, em que o objetivo é estimar o número de encomendas esperadas para cada categoria nos próximos sete dias, com base em padrões históricos de consumo. A solução é construída sobre os dados acumulados nas camadas Gold do data lake, nomeadamente fact_sales, dim_category e dim_date, e é executada diariamente às 03:00 UTC através de um DAG Apache Airflow dedicado (trendmart_ml_pipeline). Os resultados das previsões são disponibilizados no dashboard analítico da plataforma, no separador ML Insights, e persistidos na tabela lake.gold.ml_demand_forecast após validação de qualidade.

Processo de preparação e engenharia de features
A preparação dos dados é realizada no módulo machine_learning/features/timeseries_features.py, em Apache Spark, seguindo as etapas descritas abaixo:
Agregação diária por categoria
Os dados de vendas são agrupados por category_id e purchase_date, calculando o número de encomendas distintas (orders_count) e a receita total (total_revenue) para cada par categoria-dia. Esta operação transforma dados ao nível de encomenda num formato de série temporal diária.
Construção de features de lag (autocorrelação temporal)
Para capturar padrões de dependência temporal, são criadas quatro variáveis de desfasamento temporal. A variável lag_1 representa o número de encomendas no dia anterior (t−1), servindo como indicador imediato da tendência recente. A variável lag_7 regista as encomendas de sete dias antes (t−7), permitindo ao modelo capturar sazonalidade semanal  por exemplo, padrões de compra distintos entre fins de semana e dias úteis que se repetem ciclicamente. De forma análoga, lag_14 incorpora o valor de catorze dias antes (t−14), capturando padrões bi-semanais que o lag semanal isolado não consegue representar. Por fim, rolling_7d_mean calcula a média móvel dos últimos sete dias, introduzindo uma variável de tendência suavizada que atenua variações abruptas e pontuais no histórico de encomendas.
Features de calendário
A partir de dim_date, são incorporadas variáveis que descrevem o contexto temporal de cada observação: dia da semana (day_of_week), indicador de fim de semana (is_weekend), mês (month) e trimestre (quarter). Estas features permitem ao modelo distinguir padrões sazonais recorrentes.
Filtragem de registos incompletos
Após a construção das features de lag, são removidas as linhas em que lag_14 é nulo — o que corresponde aos primeiros 14 dias de histórico por categoria, nos quais não existe informação suficiente para calcular todas as variáveis de desfasamento. Adicionalmente, categorias com menos de 20 dias de histórico válido são excluídas do treino, assegurando robustez mínima dos modelos.
Vetor de features final
O conjunto de features que alimenta o modelo é composto por 8 variáveis: ["lag_1", "lag_7", "lag_14", "rolling_7d_mean", "day_of_week", "is_weekend", "month", "quarter"]. A variável alvo (label) é orders_count, o número de encomendas registadas no dia corrente.

5.3 Identificação e fundamentação da técnica de análise
Técnica selecionada: Regressão Linear com features de lag (por categoria)
A técnica de análise adotada é a regressão linear (LinearRegression da biblioteca Spark MLlib), aplicada de forma independente a cada categoria de produto. Esta abordagem designada por modelos per-category permite capturar dinâmicas de procura específicas de cada categoria, que seriam diluídas num modelo único global.
Justificação da escolha
A regressão linear foi selecionada em detrimento de outras alternativas (e.g., ARIMA, redes neuronais recorrentes, XGBoost) com base nos seguintes critérios:
Compatibilidade com o ecossistema tecnológico: O pipeline de dados da plataforma assenta integralmente em Apache Spark. A utilização de Spark MLlib assegura que o treino, avaliação inferência ocorrem no mesmo ambiente de processamento distribuído, sem dependências externas ou conversões de formato.
Interpretabilidade: A regressão linear produz coeficientes diretamente interpretáveis, permitindo compreender a contribuição de cada feature (e.g., o peso do lag semanal vs. o dia da semana) na previsão. Isto é valioso num contexto académico e de auditoria.
Representação implícita de séries temporais através de features de lag: Em vez de utilizar modelos de séries temporais clássicos (ARIMA, Prophet), que requerem bibliotecas externas e são mais complexos de paralelizar por categoria em Spark, a abordagem adotada transforma o problema de previsão numa tarefa de regressão supervisionada. Os padrões temporais são codificados explicitamente nas features de lag e de médias móveis, tornando o problema tratável por qualquer algoritmo de regressão.
Eficiência computacional: Com um número de categorias da ordem das dezenas (dataset Olist), o treino independente de modelos de regressão linear é extremamente rápido, adequado ao contexto de execução diária com restrições de memória (driver.memory=1g, local[2]).
Regularização L2: O parâmetro regParam=0.1 introduz regularização Ridge, penalizando coeficientes excessivamente elevados e reduzindo o risco de overfitting em categorias com poucos dados históricos.

5.4 Construção do modelo de análise




O modelo é implementado no módulo machine_learning/models/demand_forecast.py, seguindo uma arquitetura de treino por categoria. Para cada categoria com histórico suficiente (≥ 20 dias), é executado o seguinte processo:

Divisão treino/teste cronológica
Os dados são ordenados por data e divididos na proporção 80/20, preservando a ordem temporal. Esta escolha é fundamental: uma divisão aleatória introduziria data leakage, uma vez que dados futuros seriam usados para treinar o modelo, resultando numa avaliação irrealisticamente otimista.

Construção do pipeline Spark MLlib
O pipeline é composto por dois estágios encadeados, VectorAssembler (agrega as 8 features num vetor denso compatível com os algoritmos MLlib) LinearRegression (modelo de regressão com maxIter=100 e regParam=0.1)

Treino e registo no MLflow
Cada modelo treinado é registado no MLflow com o ID de experiência demand_forecasting, incluindo os parâmetros (category, algorithm, reg_param), as métricas de avaliação (rmse, mae) e o artefacto completo do pipeline Spark serializado em MinIO.

Geração de previsões para os próximos 7 dias
Para cada categoria, são construídas 7 linhas de previsão (D+1 a D+7), com as respetivas features de lag calculadas a partir dos últimos valores históricos conhecidos, e as features de calendário derivadas diretamente das datas futuras. 

A primeira tarefa executa o treino, avaliação e escrita na sandbox, passando as métricas agregadas via XCom. A segunda tarefa valida essas métricas contra os limiares configurados e, em caso de aprovação, promove os dados da sandbox para a tabela Gold de produção.





5.5 Validação do desempenho do modelo




O desempenho de cada modelo por categoria é avaliado no conjunto de teste (os 20% mais recentes de cada série temporal) através de duas métricas complementares: O RMSE penaliza de forma quadrática os erros de maior magnitude, tornando-o sensível a previsões com desvios elevados em dias de pico ou de rutura atípica. É a métrica primária de controlo de qualidade. O MAE representa o erro médio absoluto em unidades de encomendas, sendo mais robusto a outliers e de interpretação direta: um MAE de 5 significa que, em média, o modelo erra por 5 encomendas por dia. O sistema implementa um padrão de promoção condicional (Analytical Sandbox): os modelos treinados são primeiro escritos numa zona de sandbox (lake.sandbox) e só são promovidos para produção (lake.gold) se as métricas médias, calculadas sobre todas as categorias, cumprirem simultaneamente os limiares configurados: RMSE médio ≤ 50.0 encomenda, MAE médio≤ 40.0 encomenda
Se qualquer um dos limiares for excedido, a tarefa promote_to_gold falha com um ValueError, o Airflow marca a execução como falhada, e a tabela Gold retém os dados da execução anterior  garantindo que previsões de qualidade inferior nunca chegam ao dashboard de produção.
Rastreabilidade com MLflow
Toda a informação de treino e avaliação é registada no MLflow, acessível em http://localhost:5001, na experiência demand_forecasting. Por cada execução do DAG, existe um run MLflow por categoria, contendo parâmetros, métricas e o artefacto completo do modelo. Esta rastreabilidade permite comparações históricas de desempenho e auditoria do comportamento dos modelos ao longo do tempo.





5.6 Avaliação dos resultados

As previsões geradas pelo sistema têm reflexo direto em dois domínios operacionais: Visibilidade analítica no dashboard, O separador ML Insights do dashboard Dash apresenta, para cada categoria, a curva de procura prevista para os próximos sete dias, acompanhada dos valores de RMSE e MAE correspondentes ao modelo utilizado. Este painel permite que gestores e analistas identifiquem categorias com picos de procura antecipados e tomem decisões informadas sobre inventário e campanhas promocionais, e fundamento para alertas a vendedores: A disponibilização de previsões por categoria na tabela lake.gold.ml_demand_forecast constitui a base para a implementação de alertas automáticos a vendedores cuja oferta possa ser insuficiente face à procura prevista, uma funcionalidade de valor elevado em plataformas de marketplace.
A abordagem adotada apresenta algumas limitações que importa reconhecer. Em primeiro lugar, a regressão linear assume uma relação linear entre as features e a variável alvo, o que significa que padrões de procura não-lineares como o efeito de promoções ou uma sazonalidade abrupta não são adequadamente capturados sem transformações adicionais. Além disso, verifica-se uma ausência de variáveis exógenas, uma vez que o modelo não incorpora sinais externos como feriados, campanhas de marketing ou dados de concorrentes, que podem ter um impacto significativo na procura de determinadas categorias. Outro ponto a considerar é o horizonte de previsão fixo, visto que a geração das features de lag para os dias D+2 a D+7 baseia-se nos últimos valores históricos conhecidos, e não nas previsões dos dias anteriores do mesmo horizonte. Embora este esquema simplifique a implementação, pode acumular erro à medida que o horizonte aumenta.





6. Conclusões e Trabalho Futuro
6.1 Conclusões
O TrendMart constitui um sistema de análise de dados de grande escala funcional, desenvolvido de raiz com ferramentas de referência na indústria. A adoção de Apache Kafka e Debezium para ingestão por Change Data Capture, Apache Spark para processamento distribuído, Apache Iceberg como formato de tabela transacional, Apache Airflow para orquestração e Trino para exposição analítica representa uma stack tecnicamente madura e amplamente utilizada em ambientes de produção reais. A escolha destas ferramentas não foi arbitrária — cada uma responde a um requisito concreto da arquitetura e em conjunto permitem construir um Lakehouse robusto, escalável e auditável.
O sistema satisfaz os 22 requisitos funcionais levantados para os seis perfis de utilizador, todos validados contra a implementação real na secção 2.3. A separação de responsabilidades entre equipas, formalizada através de contratos de entrada e saída, garantiu que cada camada da arquitetura fosse desenvolvida de forma independente e coerente.
As principais limitações identificadas decorrem das condições de desenvolvimento. A fonte de dados é um simulador sintético baseado no dataset Olist, o que significa que o sistema nunca foi exposto a tráfego real de produção, com a variabilidade e irregularidade que lhe são inerentes. O Apache Spark foi executado em modo local em vez de num cluster distribuído, o que limita a capacidade de processamento e impede a validação do comportamento do sistema sob carga real. Por consequência, o volume de dados processado ao longo do projeto foi reduzido, não permitindo testar os limites de escalabilidade da arquitetura.

O TrendMart constitui um sistema de análise de dados de grande escala funcional, desenvolvido de raiz com ferramentas de referência na indústria. A adoção de Apache Kafka e Debezium para ingestão por Change Data Capture, Apache Spark para processamento distribuído, Apache Iceberg como formato de tabela transacional, Apache Airflow para orquestração e Trino para exposição analítica representa uma stack tecnicamente madura e amplamente utilizada em ambientes de produção reais. A escolha destas ferramentas não foi arbitrária — cada uma responde a um requisito concreto da arquitetura e em conjunto permitem construir um Lakehouse robusto, escalável e auditável.
O sistema satisfaz os 22 requisitos funcionais levantados para os seis perfis de utilizador, todos validados contra a implementação real na secção 2.3. A separação de responsabilidades entre equipas, formalizada através de contratos de entrada e saída, garantiu que cada camada da arquitetura fosse desenvolvida de forma independente e coerente.
As principais limitações identificadas decorrem das condições de desenvolvimento. A fonte de dados é um simulador sintético baseado no dataset Olist, o que significa que o sistema nunca foi exposto a tráfego real de produção, com a variabilidade e irregularidade que lhe são inerentes. O Apache Spark foi executado em modo local em vez de num cluster distribuído, o que limita a capacidade de processamento e impede a validação do comportamento do sistema sob carga real. Por consequência, o volume de dados processado ao longo do projeto foi reduzido, não permitindo testar os limites de escalabilidade da arquitetura.

6.2 Trabalho Futuro

Quatro direções de desenvolvimento são identificadas como prioritárias. A primeira é a redução da latência analítica para near real-time através da adoção de Spark Structured Streaming em modo contínuo em toda a cadeia de transformação, eliminando a dependência da cadência horária do pipeline batch e aproximando o sistema de um modelo de processamento de eventos em tempo real. A segunda é o deploy em infraestrutura cloud com orquestração por Kubernetes, substituindo o Docker Compose local por um ambiente distribuído capaz de suportar volumes de dados de produção e garantir a disponibilidade e escalabilidade do sistema. A terceira é a implementação de um sistema de alertas automáticos que, ao detetar anomalias no padrão de vendas ou deterioração do sentimento de cliente, notifique proativamente os perfis relevantes sem necessidade de consulta manual do dashboard. A quarta é a aplicação de técnicas de Processamento de Linguagem Natural para análise de sentimento sobre o texto livre das avaliações — na implementação atual o sentimento é inferido exclusivamente a partir do rating numérico, ignorando a informação contida na mensagem escrita pelo cliente; a adoção de um modelo de NLP permitiria classificar o sentimento diretamente a partir do texto, detetar nuances não capturadas pelo rating e extrair tópicos recorrentes de insatisfação de forma automática.
